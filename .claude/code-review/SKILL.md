# MwjRunner 代码审查技能

本技能用于 MwjRunner 项目的提交前审查和日常质量审查。审查目标是保证项目始终围绕“Python 接口自动化测试执行引擎”建设，并严格遵守：不使用 pytest 作为核心执行/断言框架、不使用 Allure 作为报告框架、自研断言体系、自研专有测试报告。

## 何时使用

必须先使用官方 `superpowers@claude-plugins-official` 插件拉齐目标、约束和质量门禁的场景：

- 实现或调整 CLI、执行引擎、调度、断言、报告、变量、HTTP 协议等核心能力。
- 修改需求、技术方案、README、CLAUDE.md 或核心约束。
- 进行跨文件重构、提交前质量门禁、风险较高或影响范围不明确的任务。

必须使用本技能的场景：

- 每次 `git commit` 之前。
- 用户要求“审查 / review / code review / 检查质量”。
- 修改 Python 源码后。
- 修改 `doc/需求规格说明书.md`、`doc/技术方案.md`、`README.md`、`CLAUDE.md` 后。
- 新增或修改 CLI、执行引擎、断言、报告、变量、用例加载、HTTP 协议相关代码后。

## 审查输入

优先审查当前工作区变更：

```bash
git status -sb
git diff --stat
git diff
git diff --cached
```

如用户指定文件或范围，则以用户指定为准。

## 审查原则

- 审查结论必须使用简体中文输出。
- 先找阻塞问题，再看可改进项。
- 不为了“看起来高级”建议过度抽象。
- 对需求文档要检查完整性、一致性、可实施性、验收标准。
- 对代码要检查正确性、安全性、可维护性、边界条件和测试可验证性。
- 对报告、日志、错误信息必须检查敏感字段脱敏。

## MwjRunner 项目硬性约束

### 禁止项

- 不得将 pytest 作为核心执行框架。
- 不得将 pytest assertion rewriting 作为核心断言能力。
- 不得使用 Allure 作为报告框架。
- 不得让调度核心直接依赖 HTTP 具体实现。
- 不得让报告生成逻辑反向依赖执行上下文。
- 不得在日志和报告中明文输出 token、password、cookie、secret、authorization 等敏感信息。

### 必须项

- 自研用例发现、加载、校验、执行调度。
- 自研 AssertionResult / RunResult 等结构化结果模型。
- 自研 Console / JSON / HTML 报告。
- 断言通过 registry 或等价机制可扩展。
- 协议执行与调度逻辑分离。
- 执行逻辑与报告生成分离。
- CLI 必须能返回稳定退出码。

## 文档审查清单

审查 `doc/需求规格说明书.md`、`doc/技术方案.md`、`README.md` 时检查：

- 是否明确项目定位和非目标。
- 是否明确不使用 pytest / Allure。
- 是否覆盖核心流程：用例加载、变量渲染、HTTP 请求、提取、断言、结果采集、报告输出。
- 是否定义 CLI 命令、参数和退出码。
- 是否定义敏感信息脱敏范围。
- 是否定义错误类型和可诊断信息。
- 是否有可执行的里程碑和验收标准。
- README 是否与需求/技术设计保持一致。
- CLAUDE.md 是否反映当前项目约束、提交规范和技能使用要求。

## Python 代码审查清单

### 架构边界

- CLI 只负责参数解析和退出码，不包含业务执行细节。
- Engine 负责组织一次运行，不直接生成具体报告页面。
- Scheduler 负责调度、retry、fail-fast、并发，不直接写 HTTP 请求细节。
- Protocol 层负责请求执行，HTTP 只是首个内置协议。
- Assertion 层只消费响应、断言规格和上下文，并输出结构化结果。
- Reporter 只消费 RunResult，不读取执行过程中的可变上下文。

### 代码质量

- 函数职责清晰，避免一个函数同时做解析、执行、报告和日志。
- 不为单一场景创建复杂抽象。
- 网络、文件、解析等边界操作必须有清晰错误信息。
- 错误信息应包含文件、suite、case、step 等定位信息。
- 并发执行时上下文必须隔离，避免 case 间变量污染。

### 安全与可靠性

- URL、路径、正则、模板渲染等用户可控输入必须谨慎处理。
- HTML 报告中展示响应体、错误信息、变量时必须转义，避免 XSS。
- 日志和报告输出前必须脱敏，而不是仅在前端隐藏。
- HTTP 请求必须支持超时，避免无限等待。
- 重试必须有上限，避免失控循环。
- 大响应体应支持截断展示或大小限制。

### 依赖审查

允许优先考虑：

- `httpx`：HTTP 客户端。
- `PyYAML`：YAML 解析。
- `jsonpath-ng`：JSONPath。
- `jsonschema`：Schema 断言。
- `Jinja2`：HTML 报告模板。
- `typer` 或 `argparse`：CLI。

新增依赖必须说明用途，通过 UV 管理，不能引入 pytest / Allure 作为核心能力。

## 工具检查建议

依赖和命令执行统一使用 UV，例如：

```bash
uv add httpx PyYAML
uv sync
uv run mwjrunner --help
```

如果项目已配置 Ruff，优先运行：

```bash
uv run ruff check .
uv run ruff format --check .
```

如果未配置 Ruff，则不强制新增；可进行人工审查并建议后续配置。

不要在用户未要求时自动提交代码。

### P0 阻塞问题
- 无 / 列表

### P1 高优先级问题
- 无 / 列表

### P2 中优先级问题
- 无 / 列表

### P3 建议项
- 无 / 列表

- Allure 禁用：通过 / 不通过

涉及具体代码时，使用 `file_path:line_number` 格式引用。

## 提交前审查要求

提交前必须完成：

1. 查看工作区状态和 diff。
2. 执行本技能审查。
3. 如有 Python 代码，运行可用的静态检查或说明未配置。
4. 如有测试命令，运行相关测试或说明当前项目尚未具备测试命令。
5. 只有 P0/P1 全部处理完，才允许继续提交。
