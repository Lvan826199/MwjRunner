# MwjRunner 发布质量门禁

本文档定义 1.0.x 发布前必须通过的本地质量门禁，用于把当前功能面收敛为可信产品。

## 发布边界

1.0.x 稳定能力聚焦 HTTP API 自动化主链路：

- YAML / JSON 用例加载与目录发现。
- HTTP 请求执行、变量渲染、变量提取。
- 自研断言体系、失败诊断、Console / JSON / HTML 报告。
- 敏感信息脱敏、退出码、CI 集成。

WebSocket、gRPC、平台、性能压测、分布式执行等能力继续按 Experimental 或平台能力管理，不能阻塞 HTTP 主链路发版。

## 黄金路径回归

每次发布至少覆盖以下场景：

1. 登录接口返回 token。
2. 从响应中提取 token 并在后续步骤复用。
3. 断言失败时输出 case、step、断言类型、期望值和实际值。
4. JSON 报告和 Console 输出不得泄露 password、token、cookie、secret、authorization 等敏感字段。
5. 退出码保持稳定：0=通过，1=断言失败，2=执行错误，3=引擎错误，4=质量门禁不通过。

上述场景由 `tests/integration/test_golden_path.py` 固化为集成回归。

## 本地门禁命令

默认门禁只依赖本仓库和本地示例服务：

```bash
uv run python scripts/release_check.py
```

发布前如需覆盖外部网络 smoke，再显式追加：

```bash
uv run python scripts/release_check.py --include-httpbin
```

默认门禁包含：

- `uv run ruff check .`
- `uv run pytest tests/unit tests/integration -q -p no:cacheprovider`
- `uv run mwjrunner validate examples/cases/quickstart_httpbin.yaml`

## Python 版本策略

当前 1.0.x 以 `pyproject.toml` 的 `requires-python = ">=3.13"` 为准。若要降低到 `>=3.11` 或 `>=3.12`，必须作为独立兼容性任务推进，并补充多版本验证矩阵，不能只修改版本声明。

## 平台与引擎契约

平台只能依赖引擎的稳定契约：

- CLI 参数与退出码。
- JSON 报告结构。
- 用例格式与报告脱敏规则。

平台不得反向要求引擎引入 Web 框架依赖，也不得绕过 CLI / JSON 报告直接读取执行过程内部状态。
