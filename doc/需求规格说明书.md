# MwjRunner 接口自动化测试执行引擎需求规格说明书

## 1. 项目定位

MwjRunner 是一个基于 Python 的接口自动化测试执行引擎，面向 HTTP API、业务流程接口、数据驱动测试和持续集成场景，目标是提供“用例易编写、执行可编排、断言可扩展、报告可追溯”的专有测试执行能力。

本项目不依赖 pytest 和 Allure，核心测试发现、执行调度、断言体系、结果采集和测试报告均由项目自研实现。

## 2. 参考成熟方案

MwjRunner 在需求设计上参考以下成熟方案的能力边界，不局限于 Python 生态，也不直接复刻其实现：

- HttpRunner：YAML/JSON 用例描述、配置与变量分层、参数化、提取与校验、hook 机制、CLI 执行、报告输出等接口自动化闭环能力。
- Postman / Newman：集合化接口定义、环境变量、批量执行、CLI 集成。
- Robot Framework：关键字驱动、可读性强的测试步骤、扩展库机制。
- Karate：接口测试 DSL、场景化测试、响应断言和数据驱动。
- JMeter：批量压测思路、执行统计、聚合结果，但本项目首期聚焦功能自动化。
- Playwright Test：清晰的执行模型、并发执行、失败重试、报告结构。
- REST Assured：链式断言、响应结构校验、接口测试语义表达。
- Pact：消费者驱动契约测试，用于保障服务间 API 契约兼容。
- Dredd：基于 OpenAPI / API Blueprint 的接口契约校验。
- Schemathesis：基于 OpenAPI / GraphQL Schema 的自动化用例生成和属性测试思路。
- Tavern：YAML 风格接口测试用例、阶段化执行和响应校验。
- Bruno / Insomnia：本地优先的接口集合管理、环境配置和 CLI 执行体验。
- k6：脚本化 API 执行、趋势指标和性能门槛思想，首期仅借鉴统计与阈值表达。
- mwjApiTest：同源接口自动化测试框架参考，关注无需编码的用例组织、文件/目录批量执行、请求封装、变量提取、函数化数据生成、HTML 报告、历史数据和通知能力。

### 2.1 成熟框架能力分析

#### 2.1.1 HttpRunner

HttpRunner 是 Python 生态中较成熟的接口自动化测试框架，核心特点是将接口测试用例声明为 YAML / JSON，并围绕 config、teststeps、variables、extract、validate、parameters、hooks 等概念组织一次接口测试运行。

可借鉴能力：

- 用例声明清晰：通过配置区和步骤区区分全局配置与具体请求步骤。
- 变量与参数化能力完整：支持变量定义、数据驱动、步骤间变量传递。
- 提取与断言闭环：请求后可提取响应字段，并在后续步骤复用。
- hooks 扩展：支持请求前后插入自定义逻辑，便于鉴权、签名、数据准备和清理。
- CLI 友好：通过命令行执行用例、生成报告，适合 CI 集成。

MwjRunner 借鉴边界：借鉴其“声明式用例 + 变量提取 + 参数化 + hook + CLI”的产品能力，不复用其 pytest 执行机制，也不依赖 pytest 作为核心运行框架。

#### 2.1.2 Postman / Newman

Postman / Newman 的优势在于集合化接口管理、环境变量、前置脚本、后置断言和命令行批量执行。

可借鉴能力：

- Collection / Environment 思路：接口集合与环境配置分离。
- 前置与后置动作：请求执行前准备变量，请求执行后提取和断言。
- CLI 集成：通过 Newman 在 CI 中执行集合并输出结果。
- 变量层级：全局变量、环境变量、集合变量、请求变量之间有清晰边界。

MwjRunner 借鉴边界：保留“用例集合 + 环境隔离 + CLI 批量执行”的体验，但用 YAML / JSON 文件作为首期主要载体，不依赖 Postman Collection 格式作为内部核心模型。

#### 2.1.3 Robot Framework

Robot Framework 的优势在于关键字驱动、业务可读性和库扩展机制。

可借鉴能力：

- 关键字驱动：将复杂动作封装为可读步骤，降低用例维护成本。
- 分层扩展：通过库或资源文件扩展能力。
- 可读报告：报告强调用例层级、步骤明细和失败定位。

MwjRunner 借鉴边界：首期不引入完整关键字 DSL，但应允许 step 具备清晰语义名称，并在后续支持内置动作或扩展动作。

#### 2.1.4 Karate

Karate 的优势在于面向 API 测试的 DSL、场景化组织、内置断言、数据驱动和并发执行体验。

可借鉴能力：

- Scenario 表达：围绕业务场景组织 API 流程。
- 内置断言语义：用简洁表达完成状态码、响应字段、schema、匹配关系断言。
- 数据驱动：同一场景可通过多组数据重复执行。
- 并发与报告：执行结果按场景聚合，方便定位失败。

MwjRunner 借鉴边界：借鉴“场景化 case + 简洁断言语义 + 数据驱动”的体验，但不引入 Gherkin 作为首期必选 DSL。

#### 2.1.5 JMeter

JMeter 更偏性能测试，但在批量执行、聚合统计、采样结果和报告指标方面成熟。

可借鉴能力：

- 聚合统计：总数、通过率、失败率、耗时分布等指标清晰。
- 采样结果：每次请求都有完整采样记录。
- 批量执行：适合较大规模接口集合运行。

MwjRunner 借鉴边界：首期不建设性能压测平台，但报告中应保留耗时、通过率、失败分类和请求响应采样信息。

#### 2.1.6 Playwright API Testing

Playwright Test 的 API 测试能力强调统一执行模型、fixture、并发、重试、trace/report 体验。

可借鉴能力：

- 稳定执行模型：测试发现、过滤、并发、重试、退出码统一。
- 上下文隔离：并发执行时测试间状态隔离。
- 报告结构清晰：失败信息、步骤、附件、耗时可追溯。

MwjRunner 借鉴边界：借鉴执行调度、上下文隔离和报告可追溯能力，不引入 Playwright 作为 API 执行依赖。

#### 2.1.7 REST Assured

REST Assured 是 Java 生态中常见的接口测试库，优势是链式请求和链式断言表达。

可借鉴能力：

- 请求语义清晰：given / when / then 风格区分准备、执行和校验。
- 断言表达丰富：状态码、header、body path、schema 等断言自然组合。
- 响应结构校验：强调响应字段路径和期望值的清晰表达。

MwjRunner 借鉴边界：借鉴其断言语义和响应路径表达，不采用 Java 链式 API 作为 MwjRunner 的主要用例形态。

#### 2.1.8 Pact

Pact 的优势在于消费者驱动契约测试，重点不是单次接口调用是否成功，而是服务消费者与服务提供者之间的 API 契约是否持续兼容。

可借鉴能力：

- 契约优先：将请求、响应和字段约束沉淀为可验证契约。
- Provider 验证：服务端变更后可验证是否破坏消费者预期。
- 版本与兼容性意识：契约可作为微服务协作和 CI 质量门禁依据。

MwjRunner 借鉴边界：首期不实现完整 Pact Broker 或消费者驱动契约平台，但应在需求中保留 OpenAPI / 契约校验能力方向，并让 JSON 报告结构适合 CI 质量门禁。

#### 2.1.9 Dredd

Dredd 的优势在于根据 API 描述文档校验真实接口实现，强调“文档即契约”。

可借鉴能力：

- OpenAPI / API Blueprint 驱动：从接口规范生成校验依据。
- 文档一致性检查：验证接口实现是否符合文档定义。
- CI 契约校验：适合在接口变更后做自动化回归。

MwjRunner 借鉴边界：首期不以 OpenAPI 驱动为唯一用例来源，但应支持后续从 OpenAPI 导入或生成基础用例，并提供契约校验扩展点。

#### 2.1.10 Schemathesis

Schemathesis 的优势在于基于 OpenAPI / GraphQL Schema 自动生成测试数据和边界场景，属于属性测试和契约测试思路。

可借鉴能力：

- Schema 驱动用例生成：从接口 Schema 自动生成正常和异常输入。
- 边界探索：覆盖枚举、类型、必填、范围、格式等边界条件。
- 反例报告：失败时记录可复现的请求样本。

MwjRunner 借鉴边界：首期不实现完整属性测试引擎，但应在后续里程碑中预留 Schema 驱动用例生成、反例固化和边界测试能力。

#### 2.1.11 Tavern

Tavern 是 Python 生态中 YAML 风格的接口测试工具，强调用例可读性、阶段化执行和响应校验。

可借鉴能力：

- YAML 用例天然可读，适合测试工程师维护。
- stage / step 结构清晰，适合描述多步骤业务流程。
- 响应校验靠近请求定义，便于定位失败。

MwjRunner 借鉴边界：借鉴 YAML 可读性和阶段化组织方式，但继续坚持自研执行、断言和报告模型。

#### 2.1.12 Bruno / Insomnia

Bruno 和 Insomnia 的优势在于接口集合管理、环境变量、本地文件化协作和调试体验。

可借鉴能力：

- 本地优先：接口集合和环境配置可直接进入版本库。
- 调试友好：接口请求、环境切换、变量查看体验清晰。
- CLI 执行：接口集合可进入 CI 流水线。

MwjRunner 借鉴边界：MwjRunner 首期不做 GUI 客户端，但项目结构、示例用例和报告路径应便于版本管理、团队协作和本地调试。

#### 2.1.13 k6

k6 更偏性能和负载测试，但它在脚本化执行、指标采集、阈值判断和 CI 门禁方面成熟。

可借鉴能力：

- 指标体系：响应时间、失败率、吞吐等指标可用于质量门禁。
- thresholds：用阈值表达是否通过质量标准。
- 场景执行：不同执行场景可配置不同策略。

MwjRunner 借鉴边界：首期不做负载测试工具，但可借鉴其阈值思想，为 response_time、失败率、错误率等指标提供质量门禁表达。

#### 2.1.14 mwjApiTest

mwjApiTest 是一个同源接口自动化测试框架，项目定位为“无需编码且易用于二次开发的接口测试框架”，并明确可拆解用于自研接口自动化平台项目的接口执行引擎。它更接近 MwjRunner 的业务语境，因此适合作为产品与架构参考。

可借鉴能力：

- 无需编码执行体验：通过示例目录中的 YAML / JSON / Python 用例组织接口测试，支持按文件或文件夹批量执行。
- 用例集合结构：用 `testSet` 描述 host、headers、interface、method、hook、verification 和 Cases，形成“公共配置 + 多条用例数据”的组织方式。
- 变量与函数化数据生成：支持 `${{var}}` 变量引用、`F{func(...)}` 工具函数调用，以及响应结果提取后写入全局或局部变量。
- 请求执行闭环：围绕请求数据处理、HTTP 请求发送、响应提取、断言校验、数据库前后置校验形成完整执行流程。
- Hook 能力：包含测试集前后置、用例前后置、请求前后置、数据库校验 hook 等扩展点。
- 报告与通知：支持 HTML 报告、历史执行数据、通过率统计、失败/错误详情，以及邮件、钉钉、企业微信通知。
- 并发与重试：支持多线程执行、失败或错误重运行，并在报告中汇总执行结果。

MwjRunner 借鉴边界：只借鉴其产品体验和架构能力，不直接复用实现。MwjRunner 不采用 unittest 作为核心执行框架，不使用 `exec` 执行用例脚本，不使用 `eval` 解析变量或函数表达式；所有变量渲染、函数调用、hook、断言和报告都必须进入受控的自研模型，并遵守敏感信息脱敏和结构化结果要求。


结合成熟框架能力，MwjRunner 需求侧必须进一步强化以下方向：

1. 用例结构应明确区分配置、变量、步骤、提取、断言和 hook，避免单个字段承载过多语义。
2. 变量系统必须支持多层级覆盖、步骤间传递、数据驱动注入和敏感变量脱敏。
3. 提取器和断言必须形成闭环：提取结果可进入上下文，断言结果必须结构化落入报告。
4. CLI 必须适合 CI 使用，支持过滤、并发、重试、报告类型、退出码和用例校验。
5. 报告必须按 project / suite / case / step / attempt 层级展示，并提供聚合统计和失败定位。
6. hooks 必须覆盖执行生命周期，但首期应保持简单，避免变成复杂插件系统。
7. 并发执行必须以 case 为隔离边界，避免变量、提取值、请求上下文互相污染。
8. 应引入契约测试意识：支持后续基于 OpenAPI / Schema 校验接口实现与接口定义的一致性。
9. 应预留 Schema 驱动用例生成能力：后续可从接口规范生成正常、异常和边界输入。
10. 报告应支持质量门禁指标：例如断言失败率、请求错误率、响应时间阈值和契约校验结果。
11. 用例和配置应适合版本库协作，避免依赖只能在 GUI 中维护的隐式状态。
12. 应提供无需编码即可完成常见接口测试的基础体验，允许通过 YAML / JSON 描述公共配置、请求步骤、断言、提取和数据驱动。
13. 应支持受控的内置函数化数据生成，例如随机手机号、时间戳、哈希摘要等，但禁止通过 `eval` 或任意脚本执行实现表达式解析。
14. Hook 能力应覆盖测试集、用例、步骤和请求响应生命周期，但必须以白名单函数、注册表或等价安全机制执行，禁止直接执行不可信字符串脚本。
15. 报告应保留历史执行趋势和通知扩展点，例如邮件、钉钉、企业微信，但通知能力不进入 M1 最小闭环。
16. 首期不以兼容外部框架格式为目标，但内部模型应能承载 HttpRunner、Postman、Karate、Pact、Dredd、Schemathesis、mwjApiTest 等成熟框架或同源项目中的核心接口测试概念。

## 3. 建设目标

### 3.1 核心目标

1. 支持接口测试用例的标准化描述、加载、校验和执行。
2. 支持自研断言体系，不依赖 pytest assertion rewriting。
3. 支持自研测试报告，覆盖 HTML、JSON、控制台摘要三类输出。
4. 支持环境变量、全局变量、用例变量、动态提取和步骤间传递。
5. 支持测试套件、标签、优先级、依赖关系、重试、跳过和失败快停。
6. 支持 CI/CD 命令行运行，返回标准退出码。
7. 支持插件化扩展，包括协议适配、断言扩展、报告扩展、通知扩展。

### 3.2 非目标

1. 首期不实现完整性能压测平台。
2. 首期不实现 UI 自动化测试。
3. 首期不引入 pytest、Allure 作为核心依赖。
4. 首期不做复杂分布式调度，保留扩展接口。

## 4. 用户角色

| 角色 | 诉求 |
| --- | --- |
| 测试工程师 | 编写和维护接口测试用例，查看失败原因和历史报告 |
| 后端开发 | 在本地或 CI 中快速验证接口契约和核心流程 |
| 测试负责人 | 查看执行质量、通过率、失败分布和趋势 |
| DevOps / CI 管理员 | 将引擎接入流水线，基于退出码和报告进行质量门禁 |

## 5. 典型使用场景

1. 本地执行指定接口用例文件。
2. 按标签执行 smoke、regression、critical 用例。
3. 在 CI 中执行全量回归并产出 HTML 报告。
4. 登录接口提取 token，后续接口自动复用。
5. 从 CSV / JSON / Excel 数据文件驱动同一接口执行多组数据。
6. 失败后自动重试，并在报告中保留每次尝试的请求、响应、断言信息。
7. 接口响应字段变化时，报告精确展示断言失败路径、期望值和实际值。

## 6. 功能需求

### 6.1 用例模型

#### 6.1.1 用例组织

- 支持 project、suite、case、step 四级结构。
- suite 可包含多个 case。
- case 可包含多个 step。
- step 是最小执行单元，通常对应一次接口请求或一个内置动作。

#### 6.1.2 用例格式

推荐首期支持 YAML 和 JSON 两种格式：

- YAML：适合人工编写和阅读。
- JSON：适合平台或工具生成。

用例文件必须支持以下字段。当前 T5 阶段已先支持单个 YAML 用例文件的 `name`、`tags`、`variables`、`steps`、`request`、`extract`、`assertions` 加载和基础校验；JSON 文件、priority、suite / project 级组织和数据驱动在后续任务继续补齐。

```yaml
name: 用户登录流程
tags: [smoke, auth]
priority: P0
variables:
  username: demo
  password: "123456"
steps:
  - name: 登录成功
    request:
      method: POST
      url: /api/login
      headers:
        Content-Type: application/json
      json:
        username: ${username}
        password: ${password}
    extract:
      token: $.data.token
    assertions:
      - type: status_code
        expected: 200
      - type: json_path
        path: $.code
        expected: 0
```

### 6.2 配置管理

- 支持项目级配置文件，例如 `mwjrunner.yaml`。
- 支持环境配置，例如 dev、test、staging、prod。
- 支持 base_url、headers、timeout、proxy、verify_ssl、retry、report_dir 等配置。
- 支持环境变量覆盖配置文件。
- 支持命令行参数覆盖环境变量和配置文件。
- 配置优先级：CLI 参数 > 环境变量 > 环境配置 > 项目默认配置 > 内置默认值。

### 6.3 变量系统

变量作用域由低到高：

1. 内置变量：时间戳、UUID、随机字符串等。
2. 全局变量：项目级共享变量。
3. 环境变量：不同运行环境下的变量。
4. suite 变量。
5. case 变量。
6. step 变量。
7. extract 提取变量。
8. CLI 注入变量。

变量表达式推荐使用 `${var_name}` 格式。

必须支持：

- 字符串变量替换。
- JSON / YAML 对象中的递归替换。
- 变量缺失时给出清晰错误。
- 敏感变量在日志和报告中脱敏展示。

### 6.4 HTTP 执行能力

首期支持 HTTP / HTTPS 接口测试：

- HTTP 方法：GET、POST、PUT、PATCH、DELETE、HEAD、OPTIONS。
- 请求参数：query、headers、cookies、form、json、body、files。
- 超时控制：连接超时、读取超时、总超时。
- SSL 验证开关。
- 代理配置。
- 自动重定向开关。
- 请求和响应内容采集。
- 请求前变量渲染。
- 响应后变量提取。

### 6.5 断言体系

项目必须自研断言模块，不使用 pytest 断言能力作为核心执行机制。

#### 6.5.1 基础断言

- `status_code`：状态码等于期望值。
- `header`：响应头字段等于、包含或匹配。
- `body_contains`：响应体包含指定文本。
- `json_path`：JSONPath 字段断言。
- `schema`：JSON Schema 结构断言。
- `equals`：实际值等于期望值。
- `not_equals`：实际值不等于期望值。
- `contains`：集合或字符串包含。
- `regex`：正则匹配。
- `length`：长度断言。
- `type`：类型断言。
- `range`：数值范围断言。
- `response_time`：响应耗时阈值断言。

#### 6.5.2 断言表达式

断言结果必须包含：

- 断言类型。
- 断言目标。
- 期望值。
- 实际值。
- 是否通过。
- 失败原因。
- 数据路径，例如 `$.data.user.id`。

#### 6.5.3 软断言和硬断言

- 硬断言：失败后终止当前 step。
- 软断言：失败后继续执行后续断言，最终汇总失败。

默认推荐：step 内断言使用软断言，关键前置条件支持硬断言。

### 6.6 数据提取

支持从响应中提取变量：

- JSONPath。
- 正则表达式。
- Header。
- Cookie。
- 原始 body。

提取失败策略：

- 默认失败并终止当前 case。
- 可配置为 optional，失败时记录 warning。

### 6.7 数据驱动

支持 data 字段或外部数据文件驱动：

- 内联 list 数据。
- JSON 文件。
- CSV 文件。
- Excel 文件可作为二期能力，首期预留接口。

每组数据生成一个独立 case instance，报告中必须可区分数据索引和参数。

### 6.8 执行调度

必须支持：

- 执行单个文件。
- 执行目录下所有用例。
- 按 suite 执行。
- 按 case 名称执行。
- 按 tags 过滤执行。
- 按 priority 过滤执行。
- 失败重试。
- fail-fast。
- 跳过用例。
- 用例依赖前置检查。
- 并发执行。

并发执行要求：

- 默认串行，保证调试体验。
- 并发 case 之间变量上下文隔离。
- suite 级共享变量必须显式声明，避免隐式污染。

### 6.9 钩子机制

支持以下 hooks：

- before_project。
- after_project。
- before_suite。
- after_suite。
- before_case。
- after_case。
- before_step。
- after_step。

Hook 可用于：

- 初始化测试数据。
- 清理测试数据。
- 注入认证信息。
- 自定义日志。
- 发送通知。

### 6.10 报告系统

项目必须自研专有测试报告，不依赖 Allure。

#### 6.10.1 报告格式

首期必须支持：

1. Console Report：终端摘要。
2. JSON Report：机器可读结果。
3. HTML Report：人工查看结果。

#### 6.10.2 报告内容

报告必须包含：

- 项目名称。
- 执行环境。
- 执行开始时间、结束时间、总耗时。
- 总用例数、通过数、失败数、跳过数、错误数。
- 通过率。
- suite 维度统计。
- case 维度统计。
- step 维度明细。
- 每个请求的 method、url、headers、body。
- 每个响应的 status_code、headers、body、elapsed。
- 每条断言的期望值、实际值和失败原因。
- 重试次数和每次尝试结果。
- 失败堆栈或异常信息。
- 脱敏后的敏感字段。

#### 6.10.3 HTML 报告体验

HTML 报告推荐能力：

- 概览仪表盘。
- suite / case / step 树形导航。
- 失败优先展示。
- 请求响应折叠展示。
- 断言失败高亮。
- 标签和优先级筛选。
- 支持离线查看，首期可采用单文件 HTML。

### 6.11 日志系统

- 支持 debug、info、warning、error 级别。
- 日志输出到控制台和文件。
- 支持 request / response debug 日志。
- 敏感字段脱敏。
- 每次执行生成唯一 run_id，用于关联日志和报告。

### 6.12 CLI 命令

推荐命令形式：

```bash
mwjrunner run tests/api --env test --tags smoke --report html,json
mwjrunner run tests/api/login.yaml --case 用户登录流程
mwjrunner validate tests/api
mwjrunner init
```

CLI 必须支持：

- `run`：执行用例。
- `validate`：校验用例格式。
- `init`：初始化项目目录结构。
- `--env`：指定环境。
- `--tags`：标签过滤。
- `--exclude-tags`：排除标签。
- `--priority`：优先级过滤。
- `--workers`：并发数。
- `--retry`：失败重试次数。
- `--fail-fast`：遇失败即停止。
- `--report`：报告类型。
- `--report-dir`：报告目录。
- `--var`：命令行变量注入。

退出码：

- 0：全部通过或仅跳过。
- 1：存在失败断言。
- 2：用例加载或配置错误。
- 3：执行引擎内部错误。

### 6.13 插件机制

插件扩展点：

- AssertionPlugin：自定义断言。
- ExtractorPlugin：自定义提取器。
- ProtocolPlugin：扩展 HTTP 以外协议。
- ReporterPlugin：扩展报告格式。
- HookPlugin：扩展生命周期行为。
- DataProviderPlugin：扩展数据源。

插件加载方式推荐：

- Python entry points。
- 配置文件声明插件模块路径。

### 6.14 安全与脱敏

必须支持敏感字段脱敏：

- authorization。
- token。
- access_token。
- refresh_token。
- password。
- secret。
- cookie。
- set-cookie。

脱敏范围：

- 控制台输出。
- 日志文件。
- JSON 报告。
- HTML 报告。
- 异常信息。

### 6.15 错误处理

错误类型至少包括：

- ConfigError。
- CaseLoadError。
- ValidationError。
- VariableResolveError。
- RequestError。
- ExtractError。
- AssertionError。
- ReportError。
- PluginError。
- EngineError。

错误信息必须包含：

- 错误类型。
- 错误消息。
- 所在文件。
- suite / case / step 名称。
- 建议修复方向。

### 6.16 示例服务与使用手册

项目必须提供可本地运行的示例接口服务和面向新手的使用手册，用于支撑用户学习、示例用例编写和引擎功能验证。

#### 6.16.1 FastAPI 示例服务

`examples/api/` 已提供独立 FastAPI 示例服务，并使用 UV 单独管理环境。

示例服务必须至少覆盖以下接口：

- `GET /health`：健康检查，用于验证服务是否启动。
- `POST /api/login`：登录接口，输入 username/password，成功后返回 token。
- `GET /api/profile`：用户信息接口，需要 `Authorization: Bearer <token>`。
- `GET /api/items`：列表查询接口，支持 query 参数。
- `POST /api/items`：创建数据接口，需要 token。
- `GET /api/error`：固定错误响应，用于演示断言失败和错误报告。

示例服务应保持简单、稳定、无外部服务依赖，避免示例因外部网络或第三方系统不可用而失败。

#### 6.16.2 示例用例

`examples/cases/` 已提供与 FastAPI 示例服务联动的 YAML 用例。

示例用例必须覆盖：

- `examples/cases/health.yaml`：健康检查。
- `examples/cases/login.yaml`：登录成功、变量替换、token 提取。
- `examples/cases/login_profile.yaml`：登录后提取 token，并在后续步骤复用 token。
- `examples/cases/health_fail.yaml`：断言失败示例。

能力覆盖要求：

- 健康检查。
- 登录成功。
- 登录后提取 token。
- 后续步骤复用 token。
- status_code 断言。
- json_path 断言。
- body_contains 断言。
- 断言失败示例。
- 敏感字段脱敏示例。

#### 6.16.3 使用手册

项目必须维护 `doc/使用手册.md`。

使用手册必须面向新手，要求：

- 解释 MwjRunner 是什么、不是什么。
- 说明 Python、UV、命令行等准备工作。
- 说明如何启动 FastAPI 示例服务。
- 说明每个示例接口的用途、请求和响应。
- 说明如何编写第一个 YAML 用例。
- 说明如何运行用例。
- 说明如何查看 Console、JSON 和后续 HTML 报告。
- 说明 token 提取和变量复用。
- 说明敏感信息脱敏。
- 说明常见错误和修复方式。

使用手册中的命令和示例必须与 `examples/api/` 和 `examples/cases/` 保持一致。

### 6.17 文档驱动开发流程

项目后续开发必须通过 `doc/下一步计划.md` 管理。

要求：

- 每次开发前先阅读 `doc/下一步计划.md`。
- 如果新任务不在计划中，先更新计划，再开发。
- 每个任务必须包含目标、范围、输出物和验收标准。
- 任务完成后更新状态和实际结果。
- CLI、用例格式、示例接口、报告格式或核心约束变化时，必须同步更新使用手册、需求规格和技术方案。

### 6.18 开发测试工具边界

MwjRunner 不得使用 pytest 作为核心执行框架，也不得使用 pytest assertion rewriting 作为核心断言机制。

允许 pytest 作为项目自身开发测试工具，用于测试 MwjRunner 的 Python 模块或 `examples/api/` 示例服务。该用途不得影响 MwjRunner 用户用例的发现、执行、断言和报告链路。

## 7. 非功能需求

### 7.1 可维护性

- 核心模块职责清晰。
- 避免执行逻辑和报告逻辑耦合。
- 内部数据结构使用 dataclass 或 Pydantic 风格模型。
- 公共接口稳定，便于插件扩展。

### 7.2 可扩展性

- 协议层、断言层、报告层、数据源层必须可扩展。
- 首期聚焦 HTTP，但不能把 HTTP 逻辑写死到调度核心。

### 7.3 性能

首期目标：

- 1000 个轻量接口 step 可稳定执行。
- 报告生成不明显阻塞执行完成。
- 大响应体支持截断展示和完整落盘。

### 7.4 兼容性

- Python 版本：以当前项目 `>=3.13` 为准。
- 支持 Windows、Linux、macOS。
- 路径处理必须跨平台。

### 7.5 可观测性

- 每次执行生成 run_id。
- 日志、报告、临时文件通过 run_id 关联。
- 关键阶段输出耗时统计。

## 8. 推荐技术栈

在不使用 pytest 和 Allure 的前提下，推荐：

| 分类 | 推荐方案 | 说明 |
| --- | --- | --- |
| 依赖管理 | UV | 统一通过 `uv add`、`uv sync`、`uv run` 管理三方库和命令执行 |
| CLI | Typer 或 argparse | Typer 体验更好；如需更少依赖可用 argparse |
| HTTP 客户端 | httpx | 支持同步、异步、超时和连接池 |
| YAML 解析 | PyYAML | 用例和配置读取 |
| JSONPath | jsonpath-ng | 响应字段提取和断言 |
| Schema 校验 | jsonschema | JSON Schema 断言 |
| 模板渲染 | Jinja2 或自研轻量变量替换 | 首期推荐自研变量替换，报告可用 Jinja2 |
| HTML 报告 | Jinja2 + 静态 CSS/JS | 生成单文件报告 |
| 数据模型 | dataclasses / pydantic | 首期推荐 dataclasses 降低依赖 |
| 日志 | logging | Python 标准库 |
| 并发 | asyncio / concurrent.futures | 首期同步执行 + 线程池并发更简单 |
| 打包 | pyproject.toml | 当前项目已具备基础文件 |

## 9. 里程碑建议

### M1：最小可用执行引擎

- CLI run。
- YAML 用例加载。
- HTTP 请求执行。
- status_code、json_path、body_contains 断言。
- JSON 报告。
- Console 摘要。

### M2：变量、提取和 HTML 报告

- 环境配置。
- 变量替换。
- JSONPath / Header / Cookie 提取。
- HTML 单文件报告。
- 敏感字段脱敏。

### M3：数据驱动和调度增强

- tags / priority 过滤。
- retry / fail-fast。
- CSV / JSON 数据驱动。
- 并发执行。
- hooks。

### M4：插件化和工程化

- 插件接口。
- 自定义断言。
- 自定义报告。
- 更完整的 validate。
- 示例项目和开发者文档。

## 10. 验收标准

1. 不安装 pytest 和 Allure，也能完成用例发现、执行、断言和报告生成。
2. 能通过 CLI 执行至少一个 YAML 接口用例并输出正确退出码。
3. 断言失败时报告中能看到失败路径、期望值、实际值和原因。
4. 登录提取 token 后，后续接口可通过变量引用 token。
5. HTML 报告可离线打开并查看完整执行明细。
6. JSON 报告结构稳定，可被 CI 或平台解析。
7. 敏感字段不会明文出现在日志和报告中。
8. 用例格式错误时，能明确提示文件、字段和修复建议。
9. 提供 `examples/api/` FastAPI 示例服务和 `examples/cases/` 示例用例后，新手可按 `doc/使用手册.md` 完成启动服务、执行用例、查看报告和定位失败的完整流程。
