# MwjRunner 技术方案

## 1. 架构原则

MwjRunner 采用“加载 -> 解析 -> 调度 -> 执行 -> 采集 -> 报告”的流水线架构。核心原则：

- 执行引擎不依赖 pytest。
- 报告系统不依赖 Allure。
- HTTP 协议是首个内置协议，但核心调度不绑定 HTTP。
- 断言、提取、报告、数据源均可插件化。
- 所有执行结果统一进入内部 Result Model，再由不同 Reporter 输出。
- 用例、环境、变量、报告尽量文件化，适合进入版本库协作。
- 内部模型优先承载成熟 API 测试框架中的共性概念，例如 collection、suite、case、step、variables、extract、assertions、hooks、attempt、contract、quality gate。
- 借鉴 mwjApiTest 的无需编码、文件/目录批量执行、公共配置加多条用例数据、函数化数据生成、HTML 报告、历史数据和通知能力，但不复用其 unittest 执行模型。
- OpenAPI / Schema / 契约校验作为后续扩展方向，不作为 M1 最小闭环的强依赖。

目录设计说明：

- M1 当前已实现 `cli`、`core`、`cases`、`http`、`assertions`、`reports`、`logging`、`utils` 等基础包；`mwjrunner run` 已接入单个 YAML 用例文件的最小执行编排。
- 当前 `cases` 包已落地为 `model.py`、`errors.py`、`loader.py` 和 `__init__.py`，支持单个 YAML 用例加载和基础校验，后续再按需拆分 parser / validator。
- `examples/api/` 是仓库内独立 FastAPI 示例服务，使用 UV 单独管理环境，不应污染主引擎依赖。
- `examples/cases/` 存放与 FastAPI 示例服务联动的 MwjRunner 示例用例。
- `doc/下一步计划.md` 是后续开发任务入口；每次开发前必须先确认或更新该文件。
- `doc/使用手册.md` 面向新手，所有示例命令和 YAML 用例应与 `examples/api/` 保持一致。
- `contracts/` 用于后续 OpenAPI / Schema / 契约校验，不阻塞最小引擎。
- `importers/` 用于后续从 OpenAPI、Postman Collection 等外部描述生成 MwjRunner 用例，不作为首期核心模型。
- `quality_gate.py` 用于基于 RunResult 判断退出码和质量门禁，例如失败率、错误率、响应时间阈值、契约校验结果。
- 参考 mwjApiTest 的实践经验，M1 应优先保证 YAML 用例、HTTP 请求执行、变量替换、提取、基础断言、Console / JSON 报告和脱敏能力稳定；文件/目录批量加载、多条 case 数据展开、HTML 报告和执行历史后续逐步补齐。

## 3. 核心模块职责

### 3.1 CLI 层

负责解析命令行参数，生成 RunConfig，调用 Engine。

命令：

- `mwjrunner run`。
- `mwjrunner validate`。
- `mwjrunner init`。
- `mwjrunner inspect`：后续可用于查看用例展开结果、变量来源和配置合并结果。
- `mwjrunner import`：后续可用于从 OpenAPI / Postman Collection 生成基础用例。

CLI 只做参数解析和结果退出码处理，不包含执行逻辑。

### 3.2 Case Loader

负责：

- 读取 YAML / JSON。
- 转换为内部 Case Model。
- 校验必填字段和字段类型。
- 输出可诊断的加载错误，错误信息包含文件、字段、原因和修复建议。
- 展开数据驱动实例。
- 支持“公共配置 + 多条用例数据”的文件组织方式，例如 suite 或 case 级公共 request 配置叠加 steps / cases 中的具体数据。
- 保留源文件、suite、case、step 位置信息，便于报告和错误定位。

用例加载不依赖外部框架格式。即使后续支持 OpenAPI / Postman 导入，也应先转换为 MwjRunner 内部模型再执行。

当前 `mwjrunner/cases/` 已实现单个 YAML 用例加载和基础校验，包含 `model.py`、`errors.py`、`loader.py` 和 `__init__.py`。已落地的内部模型包括 `TestCase`、`TestStep`、`RequestSpec`、`AssertionSpec`、`ExtractSpec`，加载错误通过 `CaseLoadError` 表达。HTTP 执行、变量替换、JSONPath 提取、基础断言、Console / JSON 报告、日志、统一脱敏和 CLI run 最小执行编排已落地，当前 `mwjrunner run` 可串联这些能力执行单个 YAML 用例文件。

### 3.3 Config Loader

负责合并：

1. 内置默认值。
2. 项目配置。
3. 环境配置。
4. 系统环境变量。
5. CLI 参数。

配置模型应覆盖：

- `base_url`。
- 全局 `headers`、`cookies`。
- `timeout`、`proxy`、`verify_ssl`、`follow_redirects`。
- `retry`、`fail_fast`、`workers`。
- `report_dir`、`report_types`。
- 敏感字段规则。
- quality gate 阈值。

配置合并必须显式记录来源，便于 `mwjrunner inspect` 后续展示变量和配置覆盖关系。参考 mwjApiTest 的全局 ENV 与局部 env 思路，MwjRunner 应保留全局变量与 case 变量的边界，但禁止并发执行时隐式写共享全局状态。

### 3.4 Engine

负责一次完整运行：

1. 创建 run_id。
2. 初始化日志。
3. 加载配置。
4. 加载用例。
5. 应用过滤规则。
6. 创建隔离的 RunContext。
7. 调用 Scheduler 执行。
8. 收集 Result Model。
9. 调用 QualityGate 计算质量门禁结果。
10. 调用 Reporter 输出报告。
11. 返回退出码。

Engine 不直接发送 HTTP 请求，不直接渲染具体报告页面。

### 3.5 Scheduler

负责：

- 串行执行。
- 并发执行。
- retry。
- fail-fast。
- skip。
- case dependency 检查。
- attempt 结果采集。

调度边界：

- 默认串行执行，保证调试体验。
- 并发以 case instance 为隔离边界。
- 每个 case instance 使用独立 CaseContext。
- suite 共享变量必须显式声明，禁止并发执行时隐式写共享上下文。
- Scheduler 只调度 StepExecutor，不直接依赖 HTTP 具体实现。

### 3.6 Protocol Client

首期内置 HttpProtocolClient，统一输入 StepRequest，输出 StepResponse。

HttpProtocolClient 负责：

- method、url、query、headers、cookies、form、json、body、files。
- timeout、proxy、verify_ssl、follow_redirects。
- 采集 request / response 原始信息。
- 采集 elapsed、status_code、headers、body。
- 将网络异常转换为结构化 RequestError。
- 使用显式 `httpx.HTTPTransport()` 创建 Client，绕过系统代理设置，确保直连目标服务。

未来可扩展：

- gRPC。
- WebSocket。
- MQ。
- Dubbo。

### 3.7 Assertion Engine

负责执行断言并生成 AssertionResult。

设计要点：

- 每种断言是独立类或函数。
- 通过 registry 注册。
- step 执行时按声明顺序执行断言。
- 支持 soft / hard 模式。
- 所有断言失败都转为结构化结果，不直接中断整个进程。
- 断言结果必须记录 type、target、expected、actual、path、passed、message。

当前已内置断言：

- status_code。
- json_path。
- body_contains。
- contains，作为响应体包含断言别名。

后续扩展断言：

- header。
- response_time。
- schema。
- regex。
- equals / not_equals。
- type / length / range。
- contract。

### 3.8 Extractor Engine

负责从 StepResponse 中提取变量并写入 CaseContext。

当前已内置提取器：

- JSONPath。

后续扩展提取器：

- Header。
- Cookie。
- Regex。

设计要点：

- 提取结果必须记录来源 step、变量名、提取路径、是否成功。
- 提取失败默认终止当前 case。
- optional 提取失败只记录 warning。
- 敏感变量写入上下文后，在日志和报告输出前必须脱敏。

### 3.9 Reporter

所有报告基于同一个 RunResult 生成。

当前已落地：

- ConsoleReporter：终端摘要。
- JsonReporter：保存完整结构化结果。

后续计划：

- HtmlReporter：基于稳定的 `RunResult.to_dict()` / JSON 报告数据生成单文件可视化报告。

Reporter 不直接访问执行上下文，只消费结果模型。

### 3.10 Contract / Schema 模块

Contract / Schema 模块用于后续契约校验和 Schema 驱动能力，首期只保留架构位置和数据模型扩展点。

后续职责：

- 读取 OpenAPI / JSON Schema。
- 校验请求是否符合接口定义。
- 校验响应 status、headers、body schema。
- 从 OpenAPI 生成基础 case / step 草稿。
- 记录契约校验结果到 AssertionResult 或 ContractResult。

### 3.11 Quality Gate

QualityGate 基于 RunResult 计算最终质量门禁结果。

可支持的门禁指标：

- 是否存在失败断言。
- 是否存在执行错误。
- 断言失败率。
- 请求错误率。
- response_time 阈值。
- 契约校验失败数。

M1 只需要基于失败断言和执行错误返回稳定退出码；其他门禁指标可在 M2/M3 增强。

## 4. 关键数据模型

### 4.1 Case Model

```python
@dataclass
class TestProject:
    name: str
    suites: list[TestSuite]
    variables: dict[str, Any]

@dataclass
class TestSuite:
    name: str
    cases: list[TestCase]
    variables: dict[str, Any]
    tags: list[str]
    source_file: str | None = None

@dataclass
class TestCase:
    name: str
    steps: list[TestStep]
    variables: dict[str, Any]
    tags: list[str]
    priority: str | None = None
    skip: bool = False
    retry: int = 0

@dataclass
class TestStep:
    name: str
    request: RequestSpec
    assertions: list[AssertionSpec]
    extract: list[ExtractSpec]
    variables: dict[str, Any]
    hooks: StepHooks | None = None
```

### 4.2 Request / Response Model

```python
@dataclass
class RequestSpec:
    method: str
    url: str
    headers: dict[str, str]
    query: dict[str, Any]
    cookies: dict[str, str]
    json: Any | None = None
    data: Any | None = None
    body: str | bytes | None = None
    timeout: float | None = None

@dataclass
class StepResponse:
    status_code: int
    headers: dict[str, str]
    cookies: dict[str, str]
    body: str | bytes | dict[str, Any] | list[Any] | None
    elapsed_ms: float
    request: RequestSnapshot
```

### 4.3 Assertion / Extract Model

```python
@dataclass
class AssertionSpec:
    type: str
    target: str | None
    expected: Any
    mode: str = "soft"

@dataclass
class ExtractSpec:
    name: str
    type: str
    path: str
    optional: bool = False
```

### 4.4 Result Model

```python
@dataclass
class RunResult:
    run_id: str
    started_at: datetime
    ended_at: datetime | None
    summary: Summary
    cases: list[CaseResult]
    errors: list[str]

@dataclass
class CaseResult:
    name: str
    status: str
    tags: list[str]
    steps: list[StepResult]
    errors: list[str]

@dataclass
class StepResult:
    name: str
    status: str
    request: HttpRequest | None
    response: HttpResponse | None
    assertions: list[AssertionResult]
    extracts: list[ExtractResult]
    errors: list[str]

@dataclass
class AssertionResult:
    type: str
    passed: bool
    expected: Any
    actual: Any
    path: str | None
    target: str | None
    mode: str
    message: str
```

## 5. 执行流程

```text
CLI
  -> load config
  -> discover case files
  -> parse and validate cases
  -> expand data-driven cases
  -> filter by tags / priority / name
  -> create run context
  -> before_project hooks
  -> execute suites
      -> before_suite hooks
      -> execute cases
          -> create isolated case context
          -> before_case hooks
          -> execute steps
              -> before_step hooks
              -> render variables
              -> send request by protocol client
              -> collect response snapshot
              -> extract variables
              -> run assertions
              -> collect attempt result
              -> after_step hooks
          -> after_case hooks
      -> after_suite hooks
  -> after_project hooks
  -> build run result
  -> apply quality gate
  -> mask sensitive fields
  -> generate reports
  -> return exit code
```

## 6. 断言实现建议

断言统一接口：

```python
class Assertion:
    type: str

    def assert_value(self, response: StepResponse, spec: AssertionSpec, context: CaseContext) -> AssertionResult:
        ...
```

注册表：

```python
registry.register("status_code", StatusCodeAssertion())
registry.register("json_path", JsonPathAssertion())
registry.register("body_contains", BodyContainsAssertion())
```

断言不抛出业务失败异常，而是返回 AssertionResult。只有断言配置错误、路径解析错误等属于执行错误。

断言扩展原则：

- 内置断言覆盖高频 HTTP API 校验。
- Schema / contract 等断言作为增强能力，不阻塞 M1。
- 断言结果必须可序列化，便于 JSON 报告和 CI 解析。
- 断言实际值、期望值包含敏感字段时，报告输出前必须脱敏。

## 7. 变量、Hook 和数据驱动

### 7.1 变量系统

变量解析顺序：

1. 内置变量。
2. 全局变量。
3. 环境变量。
4. suite 变量。
5. case 变量。
6. step 变量。
7. extract 提取变量。
8. CLI 注入变量。

变量渲染要求：

- 支持字符串 `${var_name}` 替换。
- 支持 YAML / JSON 对象递归替换。
- 变量缺失时抛出 VariableResolveError。
- 敏感变量只在内部上下文保留原值，输出前统一脱敏。

### 7.2 Hook

Hook 生命周期：

- before_project / after_project。
- before_suite / after_suite。
- before_case / after_case。
- before_step / after_step。

首期 Hook 应保持简单：

- 支持配置内置动作或 Python 函数路径。
- Hook 失败必须结构化记录。
- Hook 不应直接修改其他 case 的上下文。

### 7.3 数据驱动

数据源：

- 内联 list。
- JSON 文件。
- CSV 文件。
- Excel 作为后续能力。

展开策略：

- 每组数据生成独立 case instance。
- 报告中记录数据索引和参数摘要。
- 并发执行时每个 case instance 独立上下文。

### 7.4 函数化数据生成

参考 mwjApiTest 的 `F{func(...)}` 数据生成体验，MwjRunner 后续可支持受控的函数化数据生成能力。

设计边界：

- 函数必须来自内置函数集合或显式注册表。
- 表达式解析只允许有限语法，例如 `${func.random_phone()}` 或等价形式。
- 禁止使用 `eval` 执行表达式。
- 禁止执行任意字符串脚本。
- 函数执行结果应进入变量渲染链路，并可在报告中脱敏展示。


## 8. 报告设计

### 8.1 JSON 报告

JSON 报告应尽量完整，作为其他平台集成的数据源。

推荐文件名：

```text
reports/{run_id}/result.json
```

当前 JSON 报告已包含：

- run_id、started_at、ended_at。
- summary。
- cases / steps。
- request / response 快照。
- assertions。
- extracts。
- errors。
- 脱敏后的敏感字段。

后续随调度增强再补齐：

- env。
- suites。
- attempts / retry 明细。
- quality_gate。
- 数据驱动实例信息。

### 8.2 HTML 报告

HTML 报告采用 **template.html + JSON 注入** 方案：

- `mwjrunner/reports/template.html`：独立 HTML 模板文件，包含完整 CSS 样式和 JavaScript 渲染逻辑。
- `mwjrunner/reports/_html_template.py`：读取模板文件，将 `"__REPORT_DATA_PLACEHOLDER__"` 占位符替换为实际 JSON 数据。
- 生成的报告为单文件，可离线打开，无外部依赖。
- 模板直接在浏览器打开时显示 mock 演示数据，方便前端调试。
- JSON 注入时对 `</script>` 进行转义（`<\/script>`），防止 HTML 结构被破坏。
- 占位符替换限制为第一次匹配（`replace(..., 1)`），避免 JSON 数据中恰好包含占位符字符串时的意外替换。

```text
reports/{run_id}/report.html
```

当前页面区域：

1. 顶部运行概览。
2. 统计卡片。
3. 用例列表。
4. case / step 折叠树。
5. 请求响应详情。
6. 断言结果表。
7. 提取变量结果。
8. 错误明细。

当前实现特性：

- 单文件离线查看。
- CSS 和 JS 内联，模板文件可独立调试。
- 使用 JavaScript 动态渲染 DOM，数据通过 JSON 占位符注入。
- 失败或错误用例默认展开，通过用例默认折叠。
- JSON body 自动格式化缩进。
- 无数据时显示 mock 演示数据和提示 banner。
- 复用 `RunResult.to_dict()` 脱敏数据源，不明文输出 token、password、cookie、secret、authorization 等敏感字段。

HTML 报告中展示响应体、错误信息、变量值时必须转义，避免 XSS。

### 8.3 Console 报告

Console 报告用于本地调试和 CI 日志，应输出：

- 总用例、通过、失败、跳过、错误。
- 总耗时。
- 失败 case / step 摘要。
- 退出码原因。

### 8.4 通知扩展

参考 mwjApiTest 的邮件、钉钉、企业微信通知能力，MwjRunner 后续可将通知作为 Reporter 扩展实现。

通知扩展应消费脱敏后的 RunResult，只发送运行摘要、报告路径、失败和错误详情，不直接读取执行上下文或原始响应对象。M1 不实现通知发送，但 Result Model 应保留通知所需字段。

### 8.5 敏感信息处理

Reporter 输出前必须调用 masking 模块处理敏感字段，不能依赖页面层隐藏。

脱敏范围：

- request headers / cookies / body。
- response headers / cookies / body。
- variables。
- extracts。
- assertion expected / actual。
- errors / traceback。

## 9. 契约与 Schema 扩展设计

契约与 Schema 能力分阶段实现：

### 9.1 M1

- 不实现完整契约测试。
- 仅保证 Result Model 能容纳契约校验结果。
- schema 断言可作为后续扩展，不阻塞最小闭环。

### 9.2 M2 / M3

- 支持 JSON Schema 响应体断言。
- 支持从 OpenAPI 导入接口基础信息。
- 支持基于 OpenAPI 校验响应 status_code、content-type、body schema。
- 报告中展示契约校验结果。

### 9.3 M4+

- 支持从 OpenAPI 生成基础用例。
- 支持 Schema 驱动边界用例生成。
- 支持失败反例固化为普通 MwjRunner 用例。
- 可评估消费者驱动契约测试能力，但不在首期实现 Pact Broker 类能力。

## 10. 依赖建议

## 11. FastAPI 示例服务设计

`examples/api/` 用于提供稳定、可本地运行的被测接口服务，支撑使用手册、示例用例和引擎开发验证。

设计原则：

- 示例服务是独立项目，使用自己的 `pyproject.toml` 和 UV 环境。
- 示例服务不依赖 MwjRunner 主引擎，避免双向耦合。
- 接口保持简单稳定，不依赖数据库、外部网络或第三方系统。
- 返回结构覆盖成功、失败、认证、列表、创建等常见接口测试场景。

建议接口：

| 接口 | 用途 | 覆盖能力 |
| --- | --- | --- |
| `GET /health` | 健康检查 | GET、status_code、json_path |
| `POST /api/login` | 登录并返回 token | JSON body、变量、提取、脱敏 |
| `GET /api/profile` | 携带 token 获取用户信息 | Header、token 复用、认证失败 |
| `GET /api/items` | 查询列表 | query、列表响应、json_path |
| `POST /api/items` | 创建数据 | POST、JSON body、认证 |
| `GET /api/error` | 固定错误响应 | 断言失败、错误报告 |

示例服务可使用 pytest 做自身开发测试，但这不改变 MwjRunner 不以 pytest 作为核心执行或断言机制的约束。

## 12. 文档驱动开发流程

后续开发必须由 `doc/下一步计划.md` 驱动。

流程：

1. 开发前阅读 `doc/下一步计划.md`。
2. 如果任务不存在，先补充任务目标、范围、输出物和验收标准。
3. 按当前任务执行，不做无关扩展。
4. 代码或行为变化后，同步更新 `doc/使用手册.md`、需求规格和技术方案。
5. 提交前执行项目代码审查流程。

`doc/使用手册.md` 必须随着 CLI、示例接口、用例格式、报告格式变化同步更新。使用手册中的示例必须来自真实 `examples/api/` 接口，避免文档和实现漂移。

## 13. 开发测试工具边界

pytest 仅允许作为开发测试工具，用于测试 MwjRunner Python 模块或 `examples/api/` 示例服务。

禁止：

- 使用 pytest 发现和执行 MwjRunner 用户用例。
- 使用 pytest assertion rewriting 作为 MwjRunner 核心断言能力。
- 使用 Allure 作为 MwjRunner 报告框架。

允许：

- 使用 pytest 编写项目自身单元测试。
- 使用 pytest 测试 FastAPI 示例服务接口。
- 在文档中明确 pytest 只是开发测试工具，不属于 MwjRunner 产品执行链路。

## 15. 风险与约束

| 风险 | 应对 |
| --- | --- |
| 自研断言能力不完整 | 先保证常用断言稳定，再通过 registry 扩展 |
| 自研报告开发成本高 | 先稳定 Console / JSON 数据结构，HTML 后续基于 JSON 渲染单文件报告 |
| 并发变量污染 | case context 隔离，suite 共享显式声明 |
| 用例 DSL 过度复杂 | 首期只支持必要字段，后续演进 |
| 错误信息不清晰 | 所有错误绑定 file/suite/case/step |
| 契约测试范围膨胀 | OpenAPI / Schema 能力后置，不阻塞 M1 |
| 外部格式兼容拖慢核心引擎 | 外部导入只转换为内部模型，不把外部格式作为执行核心 |
| 敏感信息泄露 | 日志和报告统一走 masking 模块，输出前脱敏 |

## 16. 首期最小闭环

首期完成后，应能运行如下示例：

```bash
mwjrunner run examples/cases/login_profile.yaml --base-url http://127.0.0.1:8000 --report console,json
```

并得到：

- 终端通过率摘要。
- `reports/{run_id}/result.json`。
- 正确进程退出码。
- 登录接口提取 token 后，后续 step 可以引用该变量。
- 断言失败时能看到失败路径、期望值、实际值和失败原因。
- 日志和报告中不会明文输出 token、password、cookie、secret、authorization 等敏感信息。

HTML 报告已落地，基于 JSON 报告结构渲染单文件 `reports/{run_id}/report.html`。

## 17. 与成熟框架能力的对应关系

| 成熟方案能力 | MwjRunner 技术落点 |
| --- | --- |
| HttpRunner 的 YAML/JSON、variables、extract、validate、hooks | Case Model、Variable Resolver、Extractor Engine、Assertion Engine、Lifecycle Hooks |
| Postman / Newman 的 collection、environment、CLI | project/suite/case 结构、Config Loader、CLI run |
| Karate 的 scenario、match、数据驱动 | case/step 结构、Assertion Registry、Data Provider |
| REST Assured 的 given/when/then 语义 | RequestSpec、Protocol Client、AssertionSpec 分层 |
| Robot Framework 的关键字和可读步骤 | step name、后续内置 action / hook 扩展 |
| JMeter / k6 的指标和阈值 | Summary、QualityGate、response_time 断言 |
| Pact / Dredd 的契约校验 | contracts 模块、contract assertion、OpenAPI validator |
| Schemathesis 的 Schema 驱动生成 | 后续 OpenAPI importer、Schema-based generator |
| Bruno / Insomnia 的本地文件协作 | YAML/JSON 用例、环境配置、报告目录均可版本化 |
| mwjApiTest 的无需编码、文件/目录批量执行、公共配置叠加 Cases、变量提取、函数化数据生成、HTML 报告、历史数据、通知 | Case Loader、Data Provider、Variable Resolver、Extractor Engine、Reporter、Notification Reporter 扩展 |
