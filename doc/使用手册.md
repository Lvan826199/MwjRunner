# MwjRunner 使用手册

本文档是 MwjRunner 的上手手册，目标是让刚接触接口自动化测试的用户也能照着步骤完成操作。

> 当前项目已完成 M1-M7 全部能力：FastAPI 示例服务、YAML 用例加载、HTTP 请求执行、自研断言、变量替换、JSONPath 提取、Console / JSON / HTML 报告、日志、统一脱敏、目录批量执行、CSV/JSON/Excel 数据驱动、失败重试、fail-fast、环境配置加载、tags/priority 过滤、并发执行、hooks 生命周期钩子、Postman/OpenAPI 导入、通知（钉钉/企微/飞书/Slack/邮件）、WebSocket/gRPC 协议、OAuth2 认证、报告插件接口、质量门禁、Web 管理平台（JWT 认证 + RBAC 权限 + 场景编排 + 性能基线 + CI/CD + WebSocket 推送）。

## 1. MwjRunner 是什么

MwjRunner 是一个 Python 接口自动化测试执行引擎。

你可以把它理解为一个专门用来“自动请求接口、检查接口返回结果、生成测试报告”的工具。

它主要做这些事情：

1. 读取 YAML 或 JSON 编写的接口测试用例。
2. 按步骤发送 HTTP 请求。
3. 从响应中提取变量，例如登录接口返回的 token。
4. 用自研断言检查响应是否符合预期。
5. 输出控制台摘要、JSON 报告和 HTML 报告。
6. 在报告和日志中自动隐藏 token、password、authorization 等敏感信息。

MwjRunner 不是：

- 不是 pytest 的封装。
- 不是 Allure 报告的封装。
- 不是性能压测平台。
- 不是 UI 自动化测试工具。

## 2. 适合谁使用

### 2.1 测试新手

如果你刚开始学习接口测试，可以从本文档的示例服务和 YAML 用例开始。

你不需要一开始就理解复杂框架，只需要知道：

- 接口地址是什么。
- 请求方法是 GET 还是 POST。
- 请求参数写在哪里。
- 期望响应是什么。

### 2.2 测试工程师

你可以用 MwjRunner 编写接口自动化用例，覆盖登录、查询、创建、修改、删除等接口流程。

### 2.3 后端开发

你可以在本地或 CI 中使用 MwjRunner 快速验证接口是否按预期工作。

### 2.4 CI / DevOps 维护人员

你可以根据 MwjRunner 的退出码和 JSON 报告做质量门禁。

## 3. 使用前需要准备什么

### 3.1 Python 3.13 或更高版本

MwjRunner 当前要求 Python 版本遵循 `pyproject.toml`，目前为 `>=3.13`。

检查命令：

```bash
python --version
```

预期结果类似：

```text
Python 3.13.x
```

如果版本低于 3.13，需要先安装新版本 Python。

### 3.2 UV

本项目统一使用 UV 管理依赖和运行命令。

检查命令：

```bash
uv --version
```

预期结果类似：

```text
uv 0.x.x
```

如果提示 `uv: command not found` 或类似错误，说明本机还没有安装 UV，需要先安装 UV。

### 3.3 命令行终端

你需要能打开终端并进入项目目录。

项目目录示例：

```bash
cd E:/Y_pythonProject/MwjRunner
```

说明：

- Windows 下可以使用 PowerShell、Git Bash、Windows Terminal。
- 本文档中的命令以 Git Bash / Unix 风格为主。

## 4. 项目目录说明

当前项目目录会逐步形成以下结构：

```text
MwjRunner/
  mwjrunner/              # MwjRunner 核心引擎源码，后续实现
  examples/
    api/                  # FastAPI 示例接口服务，已实现
    cases/                # MwjRunner 示例 YAML 用例，已提供
  reports/                # 执行报告输出目录，运行后生成
  doc/
    需求规格说明书.md
    技术方案.md
    下一步计划.md
    使用手册.md
  pyproject.toml
  README.md
  CLAUDE.md
```

各目录含义：

| 路径 | 作用 |
| --- | --- |
| `mwjrunner/` | MwjRunner 的核心代码，负责加载用例、执行请求、断言和报告 |
| `examples/api/` | 一个用 FastAPI 写的被测接口服务，用来给示例用例提供真实接口 |
| `examples/cases/` | 示例 YAML 用例，会请求 `examples/api/` 提供的接口 |
| `reports/` | 每次运行后生成报告，例如 `result.json` 和 `report.html` |
| `doc/` | 项目文档 |

## 5. 为什么需要 FastAPI 示例服务

接口自动化测试必须有“被测接口”。

为了让所有示例都能稳定运行，本项目会在 `examples/api/` 中提供一个独立 FastAPI 示例服务。

它的作用是：

1. 给新手提供一个不用额外准备的测试对象。
2. 让使用手册中的每个例子都能真实请求接口。
3. 方便开发 MwjRunner 时验证 HTTP 请求、变量提取、断言和报告。
4. 避免依赖外部网络接口导致示例不稳定。

## 7. FastAPI 示例接口说明

FastAPI 示例服务已在 `examples/api/` 中提供以下接口。

### 7.1 健康检查：GET /health

用途：确认服务是否启动。

请求：

```http
GET /health
```

响应示例：

```json
{
  "status": "ok"
}
```

适合测试的能力：

- GET 请求。
- 状态码断言。
- JSONPath 断言。

### 7.2 登录：POST /api/login

用途：模拟用户登录，返回 token。

请求：

```http
POST /api/login
Content-Type: application/json
```

请求体：

```json
{
  "username": "demo",
  "password": "123456"
}
```

响应示例：

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "token": "demo-token"
  }
}
```

适合测试的能力：

- POST 请求。
- JSON 请求体。
- `status_code` 断言。
- `json_path` 断言。
- 从响应中提取 token。
- password 和 token 脱敏。

### 7.3 用户信息：GET /api/profile

用途：携带 token 获取当前用户信息。

请求：

```http
GET /api/profile
Authorization: Bearer demo-token
```

响应示例：

```json
{
  "code": 0,
  "data": {
    "username": "demo",
    "nickname": "示例用户"
  }
}
```

适合测试的能力：

- Header 传参。
- 使用上一步提取出的 token。
- 多步骤接口流程。
- authorization 脱敏。

### 7.4 商品列表：GET /api/items

用途：获取商品列表，支持 query 参数。

请求示例：

```http
GET /api/items?keyword=book
```

响应示例：

```json
{
  "code": 0,
  "data": [
    {
      "id": 1,
      "name": "book",
      "price": 39.9
    }
  ]
}
```

适合测试的能力：

- Query 参数。
- 列表响应。
- JSONPath 断言。

### 7.5 创建商品：POST /api/items

用途：携带 token 创建商品。

请求示例：

```http
POST /api/items
Authorization: Bearer demo-token
Content-Type: application/json
```

请求体：

```json
{
  "name": "pen",
  "price": 9.9
}
```

响应示例：

```json

  "code": 0,
  "message": "created",
  "data": {
    "id": 2,
    "name": "pen",
    "price": 9.9
  }
}
```

适合测试的能力：

- 认证接口。
- JSON 请求体。

### 7.6 错误示例：GET /api/error

用途：固定返回错误响应，方便演示断言失败和错误报告。

响应示例：

## 8. 第一个用例：健康检查

> 本节示例文件已放在 `examples/cases/health.yaml`。

### 8.1 YAML 用例内容

```yaml
name: 健康检查
tags: [smoke]
steps:
  - name: 服务健康状态
    request:
      method: GET
      url: /health
    assertions:
      - type: status_code
        expected: 200
      - type: json_path
        path: $.status
        expected: ok
```

### 8.2 每一行是什么意思

```yaml
name: 健康检查
```

表示这个用例的名称叫“健康检查”。

```yaml
tags: [smoke]
```

表示这个用例的标签是 `smoke`。后续可以通过标签筛选执行。

```yaml
steps:
```

表示下面开始定义测试步骤。一个用例可以有多个步骤。

```yaml
- name: 服务健康状态
```

表示这个步骤叫“服务健康状态”。

```yaml
request:
  method: GET
  url: /health
```

表示这个步骤会发送一个 GET 请求，请求路径是 `/health`。

```yaml
assertions:
```

表示下面开始定义断言，也就是检查响应是否符合预期。

```yaml
- type: status_code
  expected: 200
```

表示期望接口返回状态码是 200。

```yaml
- type: json_path
  path: $.status
  expected: ok
```

表示从响应 JSON 中取 `status` 字段，期望它等于 `ok`。

如果响应是：

```json
{
  "status": "ok"
}
```

那么 `$.status` 取到的值就是 `ok`。

## 11. 第二个用例：登录并复用 token

项目还提供 `examples/cases/login.yaml`，用于单独演示登录接口、变量替换、token 提取和基础断言。

```yaml
name: 登录成功
tags: [smoke, auth]
variables:
  username: demo
  password: "123456"
steps:
  - name: 使用正确账号登录
    request:
      method: POST
      url: /api/login
      headers:
        Content-Type: application/json
      json:
        username: ${username}
        password: ${password}
    extract:
      token: $.data.token
    assertions:
      - type: status_code
        expected: 200
      - type: json_path
        path: $.code
        expected: 0
      - type: json_path
        path: $.data.token
        expected: demo-token
      - type: body_contains
        expected: success
```

下面的 `examples/cases/login_profile.yaml` 在登录基础上继续演示 token 复用。

这个示例演示接口自动化中最常见的流程：

1. 调用登录接口。
2. 从登录响应中提取 token。
3. 后续接口使用这个 token。

> 本节示例文件已放在 `examples/cases/login_profile.yaml`。

### 11.1 YAML 用例内容

```yaml
name: 登录后获取用户信息
tags: [smoke, auth]
variables:
  username: demo
  password: "123456"
steps:
  - name: 登录成功
    request:
      method: POST
      url: /api/login
      headers:
        Content-Type: application/json
      json:
        username: ${username}
        password: ${password}
    extract:
      token: $.data.token
    assertions:
      - type: status_code
        expected: 200
      - type: json_path
        path: $.code
        expected: 0
      - type: body_contains
        expected: success

  - name: 获取用户信息
    request:
      method: GET
      url: /api/profile
      headers:
        Authorization: Bearer ${token}
    assertions:
      - type: status_code
        expected: 200
      - type: json_path
        path: $.data.username
        expected: demo
      - type: body_contains
        expected: 示例用户
```

### 11.2 variables 是什么

```yaml
variables:
  username: demo
  password: "123456"
```

这里定义了两个变量：

| 变量名 | 变量值 |
| --- | --- |
| `username` | `demo` |
| `password` | `123456` |

后面可以通过 `${username}` 和 `${password}` 使用它们。

### 11.3 请求体中的变量替换

```yaml
json:
  username: ${username}
  password: ${password}
```

执行时会变成：

```json
{
  "username": "demo",
  "password": "123456"
}
```

### 11.4 extract 是什么

```yaml
extract:
  token: $.data.token
```

表示从响应 JSON 中取出 `$.data.token`，保存成变量 `token`。

如果响应是：

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "token": "demo-token"
  }
}
```

那么提取后：

```text
token = demo-token
```

### 11.5 后续步骤如何使用 token

```yaml
headers:
  Authorization: Bearer ${token}
```

执行时会变成：

```http
Authorization: Bearer demo-token
```

这样第二个接口就能通过认证。

### 11.6 为什么 token 要脱敏

`token` 通常代表登录身份。如果泄露，别人可能冒充用户访问系统。

所以 MwjRunner 的报告和日志中不能明文输出 token。

报告里应该显示成类似：

```text
Authorization: ***REDACTED***
```

或：

```json
{
  "token": "***REDACTED***"
}
```

## 13. 故意制造断言失败

断言失败是接口自动化中很常见的情况。MwjRunner 需要能告诉你：

- 哪个用例失败。
- 哪个步骤失败。
- 哪条断言失败。
- 期望值是什么。
- 实际值是什么。
- 为什么失败。

### 13.1 失败用例示例

```yaml
name: 健康检查失败示例
tags: [demo, fail]
steps:
  - name: 故意写错健康状态
    request:
      method: GET
      url: /health
    assertions:
      - type: status_code
        expected: 200
      - type: json_path
        path: $.status
        expected: down
      - type: body_contains
        expected: service unavailable
```

这里 FastAPI 实际返回：

```json
{
  "status": "ok"
}
```

但用例期望：

```text
down
```

所以这条断言应该失败。

### 13.2 运行失败用例

命令示例，当前已经可以通过 `mwjrunner run` 执行：

```bash
uv run mwjrunner run examples/cases/health_fail.yaml --base-url http://127.0.0.1:8000 --report console,json
```

预期退出码：

```text
1
```

说明：

- 退出码 `1` 表示存在断言失败。
- 这不是程序崩溃，而是接口结果不符合预期。

### 13.3 失败摘要示例

```text
MwjRunner 执行摘要
run_id: 20260509-120000
用例: 1, 通过: 0, 失败: 1, 错误: 0
步骤: 1, 通过: 0, 失败: 1, 错误: 0
断言: 3, 失败: 2
错误: 0
耗时: 120 ms

失败或错误明细
```

## 16. HTML 报告

MwjRunner 支持生成单文件 HTML 报告，适合在浏览器中查看测试结果。

### 16.1 生成 HTML 报告

在 `--report` 参数中加入 `html`：

```bash
uv run mwjrunner run examples/cases/login_profile.yaml --base-url http://127.0.0.1:8000 --report console,json,html
```

### 16.2 报告路径

```text
reports/{run_id}/report.html
```

### 16.3 HTML 报告包含内容

- 运行概览：run_id、开始/结束时间、总耗时。
- 统计卡片：用例、步骤、断言、错误的通过/失败/错误计数。
- 用例列表：可折叠展开，失败或错误用例默认展开。
- 步骤明细：请求方法/URL/Headers/Body、响应状态码/Headers/Body。
- 断言表格：类型、路径、期望值、实际值、消息。
- 提取变量表格：变量名、路径、值。
- 错误信息。
- 脱敏后的敏感字段（password、token、authorization 等显示为 `***REDACTED***`）。

### 16.4 安全说明

HTML 报告采用 template.html + JSON 注入方案。JSON 数据中的 `</script>` 已转义，不存在 HTML 注入风险。报告为单文件，CSS 和 JS 内联，无需联网，可离线打开。

直接在浏览器打开模板文件 `mwjrunner/reports/template.html` 可预览 mock 演示数据，方便前端调试。

## 17. 退出码说明

MwjRunner 的 CLI 应返回稳定退出码。

| 退出码 | 含义 | 说明 |
| --- | --- | --- |
| 0 | 全部通过 | 所有用例通过，或只有跳过用例 |
| 1 | 断言失败 | 程序正常运行，但接口结果不符合预期 |
| 2 | 用例或配置错误 | YAML 写错、字段缺失、配置错误等 |
| 3 | 引擎内部错误或能力尚未实现 | MwjRunner 自身出现未预期错误，或当前命令暂未支持 |

CI 可以根据退出码判断流水线是否通过。

## 18. 敏感信息脱敏说明

### 18.1 什么是敏感信息

敏感信息是不能随便出现在日志和报告里的内容，例如：

- password
- token
- access_token
- refresh_token
- cookie
- set-cookie
- secret
- authorization

### 18.2 为什么要脱敏

如果这些信息明文出现在报告里，别人拿到报告后可能可以：

- 冒充用户登录。
- 访问内部接口。
- 获取隐私数据。
- 操作测试环境或生产环境数据。

所以 MwjRunner 必须在输出报告前统一脱敏。

### 18.3 脱敏示例

原始请求：

```json
{
  "username": "demo",
  "password": "123456"
}
```

报告中应该显示：

```json
{
  "username": "demo",
  "password": "***REDACTED***"
}
```

原始 Header：

```http
Authorization: Bearer demo-token
```

报告中应该显示：

```http
Authorization: ***REDACTED***
```

## 19. pytest 使用边界

MwjRunner 的核心执行框架和核心断言机制不能使用 pytest。

也就是说：

- 用户写的接口测试用例不能变成 pytest 测试函数。
- MwjRunner 不能依赖 pytest 来发现和执行接口用例。
- MwjRunner 不能依赖 pytest assertion rewriting 来实现断言。
- MwjRunner 的报告不能依赖 Allure。

允许的情况：

- 项目自身开发时，可以使用 pytest 测试 MwjRunner 的 Python 代码。
- `examples/api/` FastAPI 示例服务也可以使用 pytest 做自身接口测试。

简单理解：

> pytest 可以是开发者检查代码质量的工具，但不能成为 MwjRunner 产品能力的一部分。

## 20. 常见问题

### 20.1 `uv` 命令不存在

现象：

```text
uv: command not found
```

原因：

本机没有安装 UV，或者 UV 不在 PATH 中。

处理：

- 先安装 UV。
- 重新打开终端。
- 再执行：

```bash
uv --version
```

### 20.2 Python 版本太低

现象：

```text
Python 3.10.x
```

原因：

当前项目要求 Python `>=3.13`。

处理：

- 安装 Python 3.13 或更高版本。
- 确认终端中的 `python` 指向新版本。

### 20.3 FastAPI 服务没有启动

现象：

```text
Connection refused
```

原因：

MwjRunner 请求 `http://127.0.0.1:8000`，但 FastAPI 服务没启动。

处理：

进入 `examples/api`，启动服务：

```bash
uv run uvicorn app.main:app --reload
```

### 20.4 端口 8000 被占用

现象：

```text
Address already in use
```

原因：

本机已有其他程序占用了 8000 端口。

处理方式一：关闭占用 8000 的程序。

处理方式二：换一个端口，例如 8001：

```bash
uv run uvicorn app.main:app --reload --port 8001
```

然后运行用例时也要改 base-url：

```bash
uv run mwjrunner run examples/cases/health.yaml --base-url http://127.0.0.1:8001 --report console,json
```

### 20.5 用例文件路径写错

现象：

```text
Case file not found: examples/cases/health.yaml
```

原因：

用例文件不存在，或者当前目录不对。

处理：

1. 确认你在 MwjRunner 项目根目录。
2. 确认文件路径存在。
3. 使用正确路径重新运行。

### 20.6 YAML 缩进错误

现象：

```text
YAML parse error
```

原因：

YAML 对缩进很敏感，不能随便多空格或少空格。

错误示例：

```yaml
steps:
- name: 服务健康状态
 request:
   method: GET
```

正确示例：

```yaml
steps:
  - name: 服务健康状态
    request:
      method: GET
```

处理：

- 使用两个空格缩进。
- 不要混用 Tab 和空格。
- 对照本文档示例修改。

### 20.7 断言失败怎么看

如果看到：

```text
expected: down
actual: ok
```

意思是：

- 用例期望值是 `down`。
- 接口实际返回的是 `ok`。
- 所以断言失败。

处理：

- 如果接口返回是正确的，就修改用例期望值。
- 如果用例期望是正确的，就检查接口实现。

### 20.8 token 没有提取到

可能原因：

1. 登录接口失败了。
2. JSONPath 写错了。
3. 响应结构和预期不一致。

示例：

如果响应是：

```json
{
  "data": {
    "token": "demo-token"
  }
}
```

提取路径应该是：

```yaml
extract:
  token: $.data.token
```

如果写成：

```yaml
extract:
  token: $.token
```

就会提取失败。

### 20.9 报告里为什么看不到完整 token

这是正常现象。

MwjRunner 会对 token、password、authorization 等敏感字段脱敏，避免泄露。

如果报告中看到：

```text
token: ***REDACTED***
```

说明脱敏功能正在生效。

## 25. 认证方案

### 25.1 Bearer / Basic 认证

在环境配置中声明认证方案，引擎自动注入 Authorization header：

```yaml
# envs/dev.yaml
name: dev
base_url: http://127.0.0.1:8000
auth:
  type: bearer
  token: ${API_TOKEN}
```

```yaml
# Basic 认证
auth:
  type: basic
  username: admin
  password: ${ADMIN_PASSWORD}
```

### 25.2 OAuth2 认证

支持 `client_credentials` 和 `password` 两种 grant type：

```yaml
auth:
  type: oauth2
  grant_type: client_credentials
  token_url: https://auth.example.com/oauth/token
  client_id: ${CLIENT_ID}
  client_secret: ${CLIENT_SECRET}
```

引擎会自动向 `token_url` 发起请求获取 access_token，并注入到后续请求的 Authorization header 中。

## 26. 协议扩展

### 26.1 WebSocket

MwjRunner 支持 WebSocket 协议的基础测试。在 step 中使用 `protocol: websocket`：

```yaml
name: WebSocket 测试
steps:
  - name: 连接并发送消息
    protocol: websocket
    request:
      url: ws://127.0.0.1:8000/ws
      action: send_recv
      message: '{"type": "ping"}'
      timeout: 5
    assertions:
      - type: json_path
        expression: $.type
        expected: pong
```

支持的 action：
- `send_recv`：发送消息并等待响应
- `send`：只发送不等待响应

### 26.2 gRPC

MwjRunner 支持完整的 gRPC 协议测试，包括 Unary RPC 和 Server Streaming。

```yaml
name: gRPC 测试
steps:
  - name: 调用 Unary RPC
    protocol: grpc
    request:
      target: localhost:50051
      service: helloworld.Greeter
      method: SayHello
      message:
        name: "MwjRunner"
      metadata:
        authorization: "Bearer ${token}"
      timeout: 10
      tls: false
    assertions:
      - type: json_path
        expression: $.message
        expected: "Hello MwjRunner"
```

**gRPC 特性：**
- Unary RPC 调用
- Server Streaming RPC（返回消息列表）
- 动态 Server Reflection（无需预编译 stub）
- TLS / 非 TLS 连接
- Metadata 传递
- 超时控制

**依赖安装：**

gRPC 支持需要额外安装依赖（可选）：

```bash
uv add grpcio grpcio-reflection protobuf
```

## 27. 质量门禁

质量门禁用于在 CI 中自动判断测试是否达标。

### 27.1 配置质量门禁

在 `mwjrunner.yaml` 中配置：

```yaml
quality_gate:
  max_fail_rate: 0.1       # 失败率不超过 10%
  max_error_rate: 0.05     # 错误率不超过 5%
  max_response_time: 3000  # 平均响应时间不超过 3000ms
```

### 27.2 退出码

| 退出码 | 含义 |
|--------|------|
| 0 | 全部通过 |
| 1 | 存在断言失败 |
| 2 | 用例加载或配置错误 |
| 3 | 执行引擎内部错误 |
| 4 | 质量门禁不通过 |

当质量门禁不通过时，退出码为 4，CI 流水线会自动标记为失败。

## 28. 报告插件

如果内置的 Console / JSON / HTML 报告不满足需求，可以注册自定义报告插件。

### 28.1 编写自定义报告器

```python
from mwjrunner.reports.plugin import ReporterPlugin

class MyReporter(ReporterPlugin):
    name = "my_report"

    def render(self, result):
        # result 是 RunResult 对象
        return f"通过率: {result.summary.pass_rate}%"

    def write(self, result, output_dir):
        content = self.render(result)
        output_path = output_dir / "my_report.txt"
        output_path.write_text(content)
```

### 28.2 注册插件

在项目配置中注册：

```yaml
plugins:
  reporters:
    - myproject.reporters.MyReporter
```

- 指定环境、标签过滤、并发数
- 查看执行状态（运行中/通过/失败）
- 查看执行详情（统计、控制台输出、错误信息）

### 30.4 环境配置

在"环境配置"页面可以：
- 创建多个环境（dev/staging/prod）
- 配置 base_url、认证方案、Headers、Variables
- 克隆环境快速创建相似配置
- 启用/停用环境

### 30.5 场景编排

场景编排允许将多个用例串联执行，步骤间可以传递变量。

在平台中创建场景：
1. 进入"场景编排"页面
2. 点击"新建场景"
3. 添加步骤（选择已有用例）
4. 配置步骤间变量提取和传递
5. 设置步骤延迟（可选）
6. 执行场景

**场景变量传递示例：**

```
步骤1: 登录接口 → 提取 token (JSONPath: $.data.token)
步骤2: 查询用户 → 使用 ${token} 作为 Authorization
步骤3: 修改用户 → 使用 ${token} 和 ${user_id}
```

### 30.6 性能压测

在"性能压测"页面可以：
- 创建压测任务（指定 URL、并发数、总请求数）
- 查看压测结果（P50/P90/P95/P99 延迟、RPS、成功率）
- 设置性能基线
- 对比基线检测性能回归（默认阈值 20%）

### 30.7 CI/CD 集成

在"CI/CD"页面可以：
- 创建 Pipeline（关联用例、环境、触发方式）
- 手动触发 Pipeline 执行
- 查看执行记录和通过率
- 获取 SVG 通过率徽章（可嵌入 README）
- 生成 CI 配置模板（GitHub Actions / GitLab CI / Jenkins）
- 配置 Webhook 自动触发（代码推送后自动执行测试）

**Webhook 配置步骤：**

1. 在平台创建 Pipeline
2. 复制 Webhook URL（如 `http://your-server/api/webhooks/github/{pipeline_id}`）
3. 在 GitHub/GitLab 仓库的 Settings → Webhooks 中添加该 URL
4. 选择触发事件（push / pull request）
5. 代码推送后自动触发测试执行

### 30.8 通知配置

平台支持多通道通知，执行完成后自动发送结果：

| 通道 | 消息格式 | 配置方式 |
|------|----------|----------|
| 钉钉 | Markdown | Webhook URL |
| 飞书 | Interactive Card | Webhook URL |
| Slack | Blocks | Webhook URL |
| 邮件 | HTML | SMTP 配置 |

### 30.9 实时状态推送

平台通过 WebSocket 实时推送执行状态变更，无需手动刷新页面：

- 执行创建时推送 `execution:created` 事件
- 执行完成时推送 `execution:update` 事件
- 断线自动重连，不会丢失状态更新

### 30.10 权限管理

平台采用 RBAC 权限模型：

| 角色 | 权限 |
|------|------|
| admin | 全局管理，跨团队访问所有数据 |
| member | 团队内 CRUD，只能看到本团队数据 |
| viewer | 只读，不能修改数据 |

管理员可以在"用户权限"页面管理用户和团队。

## 31. 常见问题

### Q: 执行用例时报 502 错误

Windows 下 httpx 可能自动读取系统代理。MwjRunner 已内置绕过系统代理的处理，如果仍有问题，检查环境变量中是否设置了 `HTTP_PROXY` 或 `HTTPS_PROXY`。

### Q: 变量提取失败，提示响应体为空

检查上一步请求是否成功返回了数据。提取失败时错误信息会包含 status_code 和响应体预览，方便定位问题。

### Q: gRPC 用例报错找不到 grpcio

gRPC 是可选依赖，需要手动安装：

```bash
uv add grpcio grpcio-reflection protobuf
```

### Q: 平台登录后页面空白

检查前端是否正确构建（`npm run build`），Nginx 是否配置了 `try_files $uri $uri/ /index.html`。

### Q: WebSocket 连接失败

确认 Nginx 配置了 WebSocket 代理（需要 `proxy_http_version 1.1` 和 `Upgrade` 头），详见部署手册第 3.3 节。
