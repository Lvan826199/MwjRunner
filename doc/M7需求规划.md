### T49：JWT 认证体系

**优先级：P0 | 复杂度：中 | 依赖：无**

#### 背景

当前 `app/api/users.py` 使用 `_tokens: dict[str, int] = {}` 存储 Token，存在以下问题：
- 服务重启 Token 全部失效
- 无过期机制，Token 永久有效
- 无法水平扩展（多实例不共享）
- 无 Refresh Token，用户需频繁重新登录

#### 技术选型

| 方案 | 优点 | 缺点 | 结论 |
|------|------|------|------|
| PyJWT | 轻量、API 简洁、维护活跃 | 仅 JWT 编解码 | ✅ 推荐 |
| python-jose | JOSE 全集 | 维护下降，依赖有漏洞 | ❌ |
| authlib | OAuth2/OIDC 全套 | 过重 | ❌ |

选用 **PyJWT + HS256**，密钥通过环境变量注入。

#### 详细设计

**双 Token 机制：**

| Token | 算法 | 有效期 | Payload | 用途 |
|-------|------|--------|---------|------|
| Access Token | HS256 | 30 分钟 | sub, username, role, team_id, exp, iat, jti | 业务 API 访问 |
| Refresh Token | HS256 | 7 天 | sub, exp, iat, jti | 换取新 Access Token |

**新增数据库表：**

```sql
CREATE TABLE refresh_tokens (
    id INTEGER PRIMARY KEY,
    user_id INTEGER NOT NULL,
    jti VARCHAR(64) UNIQUE NOT NULL,
    expires_at DATETIME NOT NULL,
    revoked INTEGER DEFAULT 0,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);
```

**接口设计：**

| 接口 | 方法 | 说明 |
|------|------|------|
| `/api/auth/login` | POST | 登录，返回 access_token + refresh_token |
| `/api/auth/refresh` | POST | 用 refresh_token 换新 access_token |
| `/api/auth/logout` | POST | 吊销当前 refresh_token |
| `/api/auth/logout-all` | POST | 吊销用户所有 refresh_token |
| `/api/auth/change-password` | POST | 修改密码，吊销所有 Token |

**安全策略：**
- Access Token 过期后必须用 Refresh Token 续期
- Refresh Token 单次使用后轮换（返回新 Refresh Token）
- 修改密码/角色变更时吊销所有 Token
- 并发登录控制：同一用户最多 5 个活跃 Refresh Token

#### 子任务拆分

| 子任务 | 内容 | 产出 |
|--------|------|------|
| T49-1 | PyJWT 集成 + Token 工具类 | `app/core/security.py`（encode/decode/verify） |
| T49-2 | Refresh Token 模型 + 认证 API 重写 | 数据库表 + 登录/刷新/登出接口 |
| T49-3 | 认证依赖注入 + 现有 API 接入 | `get_current_user` 改为 JWT 解析，所有路由接入 |
| T49-4 | 密码安全增强 | bcrypt 替换 SHA-256，密码强度校验 |

#### 验收标准

- [ ] 登录返回 access_token（30min）+ refresh_token（7d）
- [ ] access_token 过期后返回 401，前端用 refresh_token 自动续期
- [ ] refresh_token 过期后返回 401，跳转登录页
- [ ] 修改密码后所有设备被踢出
- [ ] 服务重启后已登录用户不受影响（Token 无状态验证）
- [ ] 单元测试覆盖 Token 生成/验证/过期/吊销场景

### T50：数据权限隔离

**优先级：P0 | 复杂度：高 | 依赖：T49**

#### 背景

当前所有业务 API 无任何权限校验：
- 用例、执行、环境、Mock、Pipeline 等资源无 team_id 关联
- 任何已登录用户可操作全部数据
- 无角色权限区分（admin/manager/member 字段存在但未使用）

#### 权限模型选型

| 方案 | 适用场景 | 结论 |
|------|----------|------|
| RBAC（基于角色） | 角色固定、权限规则清晰 | ✅ 推荐 |
| ABAC（基于属性） | 权限规则复杂、动态条件多 | ❌ 过重 |

选用 **RBAC + 资源归属**：角色决定操作权限，team_id 决定数据可见范围。

#### 角色权限矩阵

| 操作 | admin | manager | member |
|------|-------|---------|--------|
| 用户管理（CRUD） | ✅ | ❌ | ❌ |
| 团队管理（CRUD） | ✅ | ❌ | ❌ |
| 本团队用例（CRUD） | ✅ | ✅ | ✅ |
| 跨团队用例（读） | ✅ | ❌ | ❌ |
| 本团队执行（触发/查看） | ✅ | ✅ | ✅ |
| 本团队环境（CRUD） | ✅ | ✅ | ❌（只读） |
| 本团队 Pipeline（CRUD） | ✅ | ✅ | ❌（只读） |
| Mock 规则（CRUD） | ✅ | ✅ | ✅ |
| 系统设置 | ✅ | ❌ | ❌ |

#### 数据库改造

需要为以下表添加 `team_id` 字段：

| 表 | 改造方式 |
|----|----------|
| test_cases | 新增 `team_id INTEGER`，默认值从创建者继承 |
| executions | 新增 `team_id INTEGER`，从触发者继承 |
| environments | 新增 `team_id INTEGER` |
| mock_rules | 新增 `team_id INTEGER` |
| pipelines | 新增 `team_id INTEGER` |
| pipeline_runs | 通过 pipeline 关联，无需单独加 |
| benchmarks | 新增 `team_id INTEGER` |

#### 中间件设计

```python
# 权限依赖注入（伪代码）
async def require_role(*roles):
    """角色校验装饰器"""

async def team_scoped_query(model, user):
    """自动注入 team_id 过滤条件"""
    if user.role == "admin":
        return select(model)  # admin 看全部
    return select(model).where(model.team_id == user.team_id)
```

#### 子任务拆分

| 子任务 | 内容 | 产出 |
|--------|------|------|
| T50-1 | 数据库迁移 + 权限工具函数 | 所有业务表加 team_id + `app/core/permissions.py` |
| T50-2 | 用例/执行/环境 API 接入权限 | 三个核心模块的 CRUD 加上团队隔离 |
| T50-3 | Mock/Pipeline/Benchmark API 接入权限 | 剩余模块接入 |
| T50-4 | 角色校验中间件 + admin 专属接口保护 | 用户管理/团队管理仅 admin 可访问 |

#### 验收标准

- [ ] member 只能看到本团队的用例/执行/环境
- [ ] admin 可以看到所有团队的数据
- [ ] member 无法访问用户管理和团队管理接口（返回 403）
- [ ] 创建资源时自动关联当前用户的 team_id
- [ ] 跨团队访问返回 404（而非 403，避免信息泄露）
- [ ] 数据迁移脚本：已有数据分配到默认团队

### T51：前端登录与路由守卫

**优先级：P0 | 复杂度：中 | 依赖：T49**

#### 背景

当前前端直接暴露所有页面，无登录入口。用户打开即可操作，无法区分身份。

#### 登录页设计

**UI 要求：**
- 居中卡片式布局，品牌 Logo + 项目名称
- 用户名 + 密码输入框 + 登录按钮
- 支持 Enter 键提交
- 登录失败显示错误提示（用户名或密码错误）
- 加载状态（按钮 loading）

**交互流程：**
```
打开页面 → 检查 Token → 有效则跳转首页
                       → 无效/过期则显示登录页
登录成功 → 存储 Token → 跳转目标页（或首页）
```

#### Token 存储策略

| 方案 | 优点 | 缺点 | 结论 |
|------|------|------|------|
| localStorage | 简单、持久化 | XSS 可读取 | ✅ 当前阶段适用 |
| httpOnly Cookie | XSS 无法读取 | 需后端配合 Set-Cookie，CSRF 风险 | 后续升级 |

当前选用 **localStorage**，配合 Token 短过期 + Refresh 机制降低风险。

#### 路由守卫逻辑

```
白名单路由：/login
其他路由：需要认证

beforeEach:
  if 目标是 /login && 已登录 → 重定向到 /
  if 目标不在白名单 && 未登录 → 重定向到 /login?redirect=原路径
  if 已登录 → 放行
```

#### Token 自动续期

```
请求拦截器:
  if 响应 401:
    if 有 refresh_token:
      调用 /api/auth/refresh
      if 成功 → 重试原请求
      if 失败 → 清除 Token，跳转登录页
    else:
      跳转登录页
```

#### Pinia Store 设计

```typescript
// stores/auth.ts
interface AuthState {
  accessToken: string | null
  refreshToken: string | null
  user: UserInfo | null
}

// Actions: login, logout, refresh, initFromStorage
// Getters: isAuthenticated, isAdmin, isManager
```

#### 子任务拆分

| 子任务 | 内容 | 产出 |
|--------|------|------|
| T51-1 | Pinia Auth Store + Token 管理 | `stores/auth.ts` + localStorage 持久化 |
| T51-2 | 登录页 + 路由守卫 | `views/Login.vue` + router beforeEach |
| T51-3 | axios 拦截器 + 自动续期 | 401 拦截 → refresh → 重试 |
| T51-4 | 顶栏用户信息 + 权限菜单控制 | Header 显示用户名/角色，非 admin 隐藏用户管理 |

#### 验收标准

- [ ] 未登录访问任何页面自动跳转 /login
- [ ] 登录成功跳转到原目标页面（redirect 参数）
- [ ] Access Token 过期后自动续期，用户无感知
- [ ] Refresh Token 过期后跳转登录页
- [ ] 页面刷新后保持登录状态
- [ ] 非 admin 用户看不到"用户权限"菜单项
- [ ] 登录页支持 Enter 键提交、loading 状态、错误提示

### T53：执行报告可视化

**优先级：P1 | 复杂度：中 | 依赖：无**

#### 背景

当前执行详情仅展示 stdout/stderr 文本，引擎生成的 JSON 报告和 HTML 报告未在平台中可视化展示。

#### 方案对比

| 方案 | 优点 | 缺点 | 结论 |
|------|------|------|------|
| iframe 嵌入 HTML 报告 | 零开发成本，复用引擎报告 | 样式隔离、交互受限 | 作为快速方案 |
| 前端重新渲染 JSON | 完全可控、交互丰富 | 开发量大 | 主方案 |
| 混合方案 | 兼顾两者 | 维护两套 | ✅ 推荐 |

**推荐混合方案：** 默认用前端组件渲染 JSON 报告（图表化），同时提供"查看原始 HTML 报告"按钮（iframe/新窗口）。

#### 图表化需求

| 图表 | 数据来源 | 说明 |
|------|----------|------|
| 断言通过率饼图 | summary.total_assertions / failed_assertions | 整体通过率一目了然 |
| 步骤耗时瀑布图 | cases[].steps[].elapsed_ms | 定位慢接口 |
| 错误分布柱状图 | 按错误类型聚合 | 快速识别主要问题 |
| 步骤状态时间线 | cases[].steps[].status + 时间 | 执行流程可视化 |

#### 报告对比功能

- 选择两次执行记录进行 diff
- 对比维度：通过率变化、新增失败用例、耗时变化
- 高亮回归项（上次通过本次失败）

#### 子任务拆分

| 子任务 | 内容 | 产出 |
|--------|------|------|
| T53-1 | 报告 JSON 解析 + 数据转换层 | `composables/useReportData.ts` |
| T53-2 | 图表组件（饼图/瀑布图/时间线） | 可复用图表组件 |
| T53-3 | 执行详情页改造 + 报告对比 | 详情页 Tab 化（概览/图表/原始报告/对比） |

#### 验收标准

- [ ] 执行详情页展示断言通过率饼图和步骤耗时瀑布图
- [ ] 点击"查看 HTML 报告"可在新窗口打开引擎生成的报告
- [ ] 报告对比功能可选择两次执行，高亮差异项

### T54：WebSocket 实时推送

**优先级：P2 | 复杂度：高 | 依赖：T52（仪表盘需要实时刷新）**

#### 背景

当前前端依赖手动刷新或定时轮询获取状态更新。执行触发后用户需反复刷新才能看到结果。

#### 技术方案对比

| 方案 | 优点 | 缺点 | 结论 |
|------|------|------|------|
| FastAPI WebSocket | 原生支持、无额外依赖 | 需自行管理连接池 | ✅ 推荐 |
| SSE (Server-Sent Events) | 简单、单向足够 | 不支持双向通信 | 备选 |
| Socket.IO | 生态成熟、自动重连 | 引入额外依赖 | ❌ 过重 |

选用 **FastAPI 原生 WebSocket**，自行实现连接管理器。

#### 推送事件设计

| 事件类型 | 触发时机 | Payload |
|----------|----------|---------|
| `execution.started` | 执行开始 | {execution_id, case_name} |
| `execution.progress` | 步骤完成 | {execution_id, passed, failed, total} |
| `execution.finished` | 执行结束 | {execution_id, status, summary} |
| `execution.log` | stdout 输出 | {execution_id, line} |
| `worker.status` | Worker 状态变更 | {worker_id, status} |
| `pipeline.triggered` | Pipeline 触发 | {pipeline_id, run_id} |
| `stats.updated` | 统计数据变更 | {} (通知前端刷新) |

#### 连接管理

- 心跳：服务端每 30 秒发送 ping，客户端 pong
- 超时：60 秒无响应断开连接
- 重连：客户端断开后 3 秒自动重连，指数退避（3s/6s/12s/30s max）
- 多标签页：每个标签页独立连接，服务端广播
- 房间机制：按 team_id 分房间，只推送本团队事件

#### 前端集成

```typescript
// composables/useWebSocket.ts — 通用封装
// composables/useExecutionWs.ts — 执行状态订阅
// composables/useGlobalWs.ts — 全局事件（仪表盘刷新）
```

降级策略：WebSocket 连接失败时自动降级为 10 秒轮询。

#### 子任务拆分

| 子任务 | 内容 | 产出 |
|--------|------|------|
| T54-1 | 后端 WebSocket 管理器 + 路由 | `app/core/ws_manager.py` + `/ws` 端点 |
| T54-2 | 执行引擎推送集成 | 执行过程中广播事件 |
| T54-3 | 前端 WebSocket composable + 页面集成 | 实时状态更新、降级轮询 |

#### 验收标准

- [ ] 触发执行后，执行列表状态实时更新（无需手动刷新）

### T56：Webhook 自动触发

**优先级：P1 | 复杂度：中 | 依赖：T47**

#### 背景

当前 Pipeline 触发仅支持手动调用 API，无法响应 Git 事件（push/PR/tag）自动触发执行。

#### Webhook 端点设计

```
POST /api/webhooks/{pipeline_id}
Headers:
  X-Hub-Signature-256: sha256=<hmac>     # GitHub
  X-Gitlab-Token: <token>                 # GitLab
  Content-Type: application/json
```

#### 验签机制

| 平台 | 验签方式 | 实现 |
|------|----------|------|
| GitHub | HMAC-SHA256（X-Hub-Signature-256） | 用 Pipeline.webhook_secret 计算 |
| GitLab | Token 比对（X-Gitlab-Token） | 直接比较 |
| 通用 | HMAC-SHA256（自定义 Header） | 兼容其他平台 |

验签失败返回 401，不触发执行。

#### 事件过滤

| 过滤维度 | 配置方式 | 说明 |
|----------|----------|------|
| 事件类型 | Pipeline 新增字段 `webhook_events` | push, pull_request, tag |
| 分支匹配 | Pipeline 新增字段 `webhook_branches` | main, develop, release/* |
| 路径匹配 | Pipeline 新增字段 `webhook_paths` | src/**, tests/** |

过滤逻辑：事件类型 AND 分支匹配 AND 路径匹配，全部满足才触发。

#### Payload 解析

```python
# GitHub push event
{
  "ref": "refs/heads/main",
  "after": "abc123...",  # commit_sha
  "repository": {"full_name": "org/repo"},
  "commits": [{"modified": ["src/api.py"]}]
}

# GitLab push event
{
  "ref": "refs/heads/main",
  "checkout_sha": "abc123...",
  "project": {"path_with_namespace": "org/repo"}
}
```

需要统一解析为内部结构：`{branch, commit_sha, paths_changed[], event_type}`

#### 子任务拆分

| 子任务 | 内容 | 产出 |
|--------|------|------|
| T56-1 | Webhook 端点 + 验签中间件 | `app/api/webhooks.py` |
| T56-2 | Payload 解析器（GitHub/GitLab/通用） | `app/services/webhook_parser.py` |
| T56-3 | 事件过滤 + 触发执行 + Pipeline 模型扩展 | 新增过滤字段 + 自动触发逻辑 |

#### 验收标准

- [ ] GitHub push 事件正确触发对应 Pipeline
- [ ] 验签失败返回 401，不触发执行
- [ ] 分支不匹配时返回 200 但不触发（静默忽略）
- [ ] PipelineRun 记录中包含 commit_sha 和 branch
- [ ] 支持 GitLab 和通用 Webhook 格式

### T58：gRPC 协议完整支持

**优先级：P2 | 复杂度：高 | 依赖：无**

#### 背景

当前引擎 `mwjrunner/protocols/grpc_adapter.py` 仅为骨架预留，无实际功能。随着微服务架构普及，gRPC 接口测试需求日益增长。

#### Proto 文件管理

| 方案 | 说明 | 结论 |
|------|------|------|
| 预编译（protoc） | 用户提前编译 proto 为 Python 代码 | 传统方案，用户门槛高 |
| 动态加载（grpc_reflection） | 运行时从服务端获取 proto 定义 | 依赖服务端开启 reflection |
| 动态解析（grpcio-tools） | 运行时解析 .proto 文件 | ✅ 推荐，灵活且用户友好 |

选用 **动态解析**：用户只需提供 .proto 文件路径，引擎自动编译并加载。

#### 用例格式设计

```yaml
name: gRPC 用户服务测试
protocol: grpc
config:
  host: localhost:50051
  proto_path: protos/user.proto
  service: user.UserService
  tls:
    enabled: false
    cert_path: ""

steps:
  - name: 创建用户
    request:
      method: CreateUser          # RPC 方法名
      message:                    # 请求 message 字段
        name: "test_user"
        email: "test@example.com"
      metadata:                   # gRPC metadata（类似 HTTP headers）
        authorization: "Bearer ${token}"
    assertions:
      - type: grpc_status
        expected: "OK"
      - type: json_path
        path: "$.id"
        operator: "exists"
    extract:
      user_id:
        type: json_path
        path: "$.id"
```

#### 断言扩展

| 断言类型 | 说明 | 示例 |
|----------|------|------|
| grpc_status | gRPC 状态码 | OK, NOT_FOUND, PERMISSION_DENIED |
| json_path | 响应 message 字段（转 dict 后） | 复用现有 json_path 断言 |
| grpc_metadata | 响应 metadata 字段 | 类似 header 断言 |

#### 流式 RPC 支持

| 类型 | 用例语法 | 说明 |
|------|----------|------|
| Unary | 默认 | 一请求一响应 |
| Server Streaming | `stream: server` | 一请求多响应，断言最后一条或聚合 |
| Client Streaming | `stream: client`, messages 数组 | 多请求一响应 |
| Bidirectional | `stream: bidi` | 多请求多响应 |

流式 RPC 为 P2 子任务，Unary 优先实现。

#### 子任务拆分

| 子任务 | 内容 | 产出 |
|--------|------|------|
| T58-1 | Proto 动态编译 + Channel 管理 | `protocols/grpc/compiler.py`, `grpc/channel.py` |
| T58-2 | Unary RPC 执行器 + 断言扩展 | `protocols/grpc/executor.py`, `assertions/grpc_status.py` |
| T58-3 | 流式 RPC 支持 | Server/Client/Bidi streaming |
| T58-4 | gRPC 用例示例 + 文档 | `examples/cases/grpc/` + 使用手册更新 |

#### 验收标准

- [ ] 提供 .proto 文件路径即可执行 gRPC 用例，无需手动编译

### T59：场景编排

**优先级：P2 | 复杂度：高 | 依赖：无**

#### 背景

当前引擎执行模型是"用例 → 步骤"线性结构，无法表达复杂业务流程（条件分支、循环、并行、等待）。

#### 编排模型设计

| 方案 | 优点 | 缺点 | 结论 |
|------|------|------|------|
| DAG（有向无环图） | 灵活、可并行 | 用户理解成本高、YAML 表达复杂 | ❌ |
| 线性 + 控制流 | 直觉、YAML 友好 | 不支持复杂并行 | ✅ 推荐 |

选用 **线性 + 控制流关键字**，保持 YAML 用例的可读性。

#### 控制流语法

```yaml
name: 订单完整流程
type: scenario                    # 标记为场景编排用例

steps:
  - name: 登录获取 Token
    request: { method: POST, url: /auth/login, body: {...} }
    extract: { token: { type: json_path, path: "$.token" } }

  - name: 创建订单
    request: { method: POST, url: /orders, headers: { authorization: "Bearer ${token}" } }
    extract: { order_id: { type: json_path, path: "$.id" } }

  # 条件分支
  - name: 检查库存
    control: if
    condition: "${order_status} == 'pending'"
    then:
      - name: 扣减库存
        request: { method: POST, url: /inventory/deduct }
    else:
      - name: 记录异常
        request: { method: POST, url: /alerts }

  # 循环
  - name: 轮询订单状态
    control: loop
    max_iterations: 10
    interval_ms: 1000
    until: "${order_status} == 'completed'"
    steps:
      - name: 查询订单
        request: { method: GET, url: /orders/${order_id} }
        extract: { order_status: { type: json_path, path: "$.status" } }

  # 并行
  - name: 并行验证
    control: parallel
    steps:
      - name: 验证支付
        request: { method: GET, url: /payments/${order_id} }
      - name: 验证物流
        request: { method: GET, url: /shipping/${order_id} }

  # 等待
  - name: 等待异步处理
    control: wait
    duration_ms: 2000
```

#### 数据流设计

| 范围 | 说明 |
|------|------|
| 全局上下文 | 场景级变量池，所有步骤共享 |
| 步骤提取 | extract 结果写入全局上下文 |
| 条件表达式 | 支持 `==`, `!=`, `>`, `<`, `contains`, `exists` |
| 循环变量 | `${__loop_index}` 当前迭代次数 |

#### 失败策略

| 策略 | 配置 | 说明 |
|------|------|------|
| 中断 | `on_fail: abort`（默认） | 任一步骤失败则整个场景失败 |
| 继续 | `on_fail: continue` | 记录失败但继续执行后续步骤 |
| 重试 | `on_fail: retry`, `retry: 2` | 步骤级重试 |
| 降级 | `on_fail: fallback` | 执行 fallback 步骤 |

#### 子任务拆分

| 子任务 | 内容 | 产出 |
|--------|------|------|
| T59-1 | 场景模型 + 控制流解析器 | `cases/scenario.py`, `core/control_flow.py` |
| T59-2 | if/else + loop 执行器 | 条件分支和循环的运行时实现 |
| T59-3 | parallel + wait + 失败策略 | 并行执行和等待 + on_fail 处理 |
| T59-4 | 场景用例示例 + 文档 | `examples/cases/scenarios/` + 使用手册 |

#### 验收标准

- [ ] if/else 条件分支根据变量值正确选择执行路径
- [ ] loop 循环在条件满足时正确退出，超过 max_iterations 时报错
- [ ] parallel 步骤并行执行，所有完成后继续
- [ ] wait 步骤正确等待指定时间
- [ ] 步骤间变量传递正常（前步 extract → 后步使用）

**优先级：P2 | 复杂度：高 | 依赖：T46（Benchmark 模块）**

#### 背景

T46 实现了单次压测（固定并发 × 固定时长），但缺乏：

| 维度 | T46 Benchmark | T60 持续性能测试 |

| 持续稳定 | 长时间低压运行 | users, duration_s (可达数小时) |

#### 实时指标采集

每秒采样一次，记录：

```json
{
  "timestamp": "2026-05-12T10:00:01Z",
  "rps": 150,
  "active_users": 20,
  "latency_p50": 45,
  "latency_p90": 120,
  "latency_p95": 180,
  "latency_p99": 350,
  "error_rate": 0.02,
  "error_count": 3,
  "success_count": 147
}
```

存储方案：
- 短期（24h）：内存 + SQLite（时间序列表）
- 长期：聚合为分钟级/小时级摘要

#### 基线对比与告警

**基线建立：**

**退化检测：**

```
if current_p95 > baseline_p95 * 1.2:  # P95 延迟增长超过 20%

if current_error_rate > baseline_error_rate + 0.05:  # 错误率增长超过 5%

```
