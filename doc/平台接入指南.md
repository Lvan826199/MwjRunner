# 平台接入指南

本文档面向测试平台开发者，说明如何将 MwjRunner 执行引擎和 Web 管理平台集成到测试平台中。

## 接入架构概览

```
┌─────────────────────────────────────────────────┐
│                  测试平台                         │
│                                                  │
│  ┌──────────┐    ┌──────────┐    ┌──────────┐   │
│  │ 用例管理  │───▶│ 调度中心  │───▶│ 报告展示  │   │
│  └──────────┘    └──────────┘    └──────────┘   │
│                       │                          │
└───────────────────────┼──────────────────────────┘
                        │
            ┌───────────┼───────────┐
            ▼                       ▼
┌───────────────────────┐  ┌────────────────────────┐
│   MwjRunner CLI 引擎   │  │  MwjRunner 平台 API     │
│                        │  │                         │
│  输入: YAML + CLI 参数  │  │  REST API + WebSocket   │
│  输出: JSON + HTML + 退出码│  │  JWT 认证 + RBAC 权限   │
└───────────────────────┘  └────────────────────────┘
```

接入方式有两种：
1. **CLI 模式**：通过子进程调用 `mwjrunner run` 命令（轻量，适合 CI 集成）
2. **API 模式**：通过 HTTP API 调用平台后端（功能完整，适合平台对接）

## 1. 调用方式

### 1.1 CLI 命令格式

```bash
mwjrunner run <用例文件路径> [选项]
```

### 1.2 核心参数

| 参数 | 说明 | 示例 |
|------|------|------|
| `<path>` | 用例文件或目录路径 | `cases/`、`cases/login.yaml` |
| `--env` | 环境名称，对应 `envs/<name>.yaml` | `--env staging` |
| `--base-url` | HTTP 请求基础地址 | `--base-url http://api.test.com` |
| `--tags` | 仅运行指定标签，逗号分隔 | `--tags smoke,api` |
| `--exclude-tags` | 排除指定标签，逗号分隔 | `--exclude-tags slow` |
| `--priority` | 仅运行指定优先级，逗号分隔 | `--priority P0,P1` |
| `--workers` | 并发执行 worker 数量 | `--workers 4` |
| `--retry` | 失败重试次数 | `--retry 2` |
| `--fail-fast` | 遇到失败后尽快停止 | `--fail-fast` |
| `--report` | 报告类型，逗号分隔 | `--report json,html,console` |
| `--report-dir` | 报告输出目录 | `--report-dir /tmp/reports` |
| `--var KEY=VALUE` | 运行时变量，可重复 | `--var token=abc --var env=staging` |

### 1.3 平台调用示例（Python subprocess）

```python
import subprocess
import json
from pathlib import Path

def run_case(case_file: str, base_url: str, variables: dict = None, report_dir: str = None):
    """调用 MwjRunner 执行单个用例文件。"""
    cmd = ["mwjrunner", "run", case_file, "--base-url", base_url, "--report", "json,html"]

    if report_dir:
        cmd.extend(["--report-dir", report_dir])

    for key, value in (variables or {}).items():
        cmd.extend(["--var", f"{key}={value}"])

    result = subprocess.run(cmd, capture_output=True, text=True, timeout=300)

    return {
        "exit_code": result.returncode,
        "stdout": result.stdout,
        "stderr": result.stderr,
    }
```

## 2. 退出码含义

引擎通过进程退出码传递执行状态，平台必须根据退出码判断执行结果：

| 退出码 | 含义 | 平台处理建议 |
|--------|------|-------------|
| `0` | 全部通过 | 标记用例为 PASSED |
| `1` | 存在断言失败 | 标记用例为 FAILED，解析报告展示失败详情 |
| `2` | 存在执行错误（用例加载、配置、变量、网络、hook 等） | 标记用例为 ERROR，提示用户排查 |
| `3` | 引擎内部错误 | 标记为系统异常，通知平台运维 |
| `4` | 质量门禁不通过 | 标记为 FAILED 或 QUALITY_GATE_FAILED，并提示门禁原因 |

## 3. JSON 报告结构

报告文件位于 `<report-dir>/<run_id>/result.json`，`run_id` 格式为 `YYYYMMDD-HHMMSS-<8位hex>`。

### 3.1 顶层结构

```json
{
  "run_id": "20260512-090929-8e2e7767",
  "started_at": "2026-05-12T08:09:29.000000+00:00",
  "ended_at": "2026-05-12T08:09:30.000000+00:00",
  "summary": { ... },
  "cases": [ ... ],
  "errors": [ "运行级错误信息" ]
}
```

### 3.2 summary 对象

```json
{
  "total_cases": 1,
  "passed_cases": 1,
  "failed_cases": 0,
  "error_cases": 0,
  "skipped_cases": 0,
  "total_steps": 2,
  "passed_steps": 2,
  "failed_steps": 0,
  "error_steps": 0,
  "total_assertions": 6,
  "failed_assertions": 0,
  "total_errors": 0,
  "elapsed_ms": 616.1
}
```

平台可直接使用 summary 字段生成统计面板和趋势图表。

### 3.3 cases 数组

```json
{
  "name": "登录后获取用户信息",
  "source_file": "cases/login_profile.yaml",
  "status": "passed",
  "steps": [ ... ],
  "errors": []
}
```

`status` 取值：`passed` | `failed` | `error`

### 3.4 steps 数组

```json
{
  "name": "登录成功",
  "status": "passed",
  "elapsed_ms": 320.5,
  "request": {
    "method": "POST",
    "url": "http://api.example.com/api/login",
    "headers": { "Content-Type": "application/json" },
    "query": {},
    "cookies": {},
    "body": "{\"username\": \"demo\", \"password\": \"***REDACTED***\"}",
    "timeout": 30.0
  },
  "response": {
    "status_code": 200,
    "headers": { "content-type": "application/json" },
    "cookies": {},
    "body": "{\"code\": 0, \"message\": \"success\", \"data\": {\"token\": \"***REDACTED***\"}}",
    "elapsed_ms": 318.2
  },
  "assertions": [ ... ],
  "extracts": [ ... ],
  "errors": []
}
```

### 3.5 assertions 数组

```json
{
  "type": "status_code",
  "passed": true,
  "expected": 200,
  "actual": 200,
  "path": null,
  "target": null,
  "mode": "soft",
  "message": "status_code 断言通过"
}
```

断言类型（`type`）：`status_code` | `json_path` | `body_contains` | `contains` | `response_time` | `json_schema` | `regex` | `header` | `cookie`

### 3.6 extracts 数组

```json
{
  "name": "token",
  "type": "json_path",
  "path": "$.data.token",
  "value": "***REDACTED***",
  "extracted": true,
  "optional": false,
  "message": "变量提取成功"
}
```

## 4. 用例文件格式（YAML）

平台需要将用户在界面上编排的用例转换为以下 YAML 格式：

```yaml
name: 用例名称
tags: [smoke, auth]
variables:
  username: demo
  password: "123456"
steps:
  - name: 步骤名称
    request:
      method: POST
      url: /api/login
      headers:
        Content-Type: application/json
      json:
        username: ${username}
        password: ${password}
    extract:
      token: $.data.token
    assertions:
      - type: status_code
        expected: 200
      - type: json_path
        path: $.code
        expected: 0
      - type: body_contains
        expected: success

  - name: 后续步骤（可引用上一步提取的变量）
    request:
      method: GET
      url: /api/profile
      headers:
        Authorization: Bearer ${token}
    assertions:
      - type: status_code
        expected: 200
```

### 4.1 字段说明

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `name` | string | 是 | 用例名称 |
| `tags` | list[string] | 否 | 标签，用于筛选 |
| `variables` | dict | 否 | 用例级变量 |
| `steps` | list | 是 | 步骤列表 |
| `steps[].name` | string | 是 | 步骤名称 |
| `steps[].request.method` | string | 是 | HTTP 方法 |
| `steps[].request.url` | string | 是 | 请求路径（相对或绝对） |
| `steps[].request.headers` | dict | 否 | 请求头 |
| `steps[].request.json` | any | 否 | JSON 请求体 |
| `steps[].request.data` | any | 否 | Form 请求体 |
| `steps[].request.body` | string | 否 | 原始请求体 |
| `steps[].request.query` | dict | 否 | URL 查询参数 |
| `steps[].request.cookies` | dict | 否 | Cookies |
| `steps[].request.timeout` | float | 否 | 超时秒数，默认 30 |
| `steps[].extract` | dict | 否 | 变量提取，key=变量名, value=JSONPath |
| `steps[].assertions` | list | 否 | 断言列表 |

### 4.2 变量系统

- 变量通过 `${变量名}` 语法引用
- 变量来源优先级：CLI `--var` > 用例 `variables` 字段
- 步骤间变量传递：前一步 `extract` 提取的变量可在后续步骤中使用
- 变量可用于 url、headers、json、data、body 等所有请求字段

### 4.3 断言类型

| type | 说明 | 必填字段 |
|------|------|---------|
| `status_code` | HTTP 状态码断言 | `expected` |
| `json_path` | JSONPath 取值断言 | `path`, `expected` |
| `body_contains` | 响应体包含子串 | `expected` |

## 5. 敏感信息脱敏

引擎会自动对以下字段进行脱敏处理，平台无需额外处理：

- 请求/响应中的 `token`、`password`、`secret`、`authorization`、`cookie` 等敏感字段
- 脱敏后显示为 `***REDACTED***`
- 脱敏作用于 JSON 报告和 HTML 报告

平台在展示报告时可直接使用脱敏后的数据，无需二次处理。

## 6. 报告文件结构

每次执行生成独立目录：

```
<report-dir>/
└── <run_id>/
    ├── result.json    # 结构化 JSON 报告（平台解析用）
    ├── report.html    # 单文件 HTML 报告（可直接提供下载或内嵌展示）
    └── run.log        # 执行日志
```

- `result.json`：平台解析的核心数据源
- `report.html`：单文件、无外部依赖、可离线打开，适合作为附件下载或 iframe 嵌入
- `run.log`：执行过程日志，排查问题时使用

## 7. 平台接入典型流程

```
1. 用户在平台 UI 编排用例
       │
       ▼
2. 平台将用例转换为 YAML 文件写入临时目录
       │
       ▼
3. 平台调用 subprocess 执行:
   mwjrunner run /tmp/cases/xxx.yaml \
     --base-url http://target-api \
     --report json,html \
     --report-dir /tmp/reports \
     --var env=staging
       │
       ▼
4. 等待进程结束，获取退出码
       │
       ▼
5. 解析 <report-dir>/<run_id>/result.json
       │
       ▼
6. 将结果写入平台数据库，更新用例执行状态
       │
       ▼
7. 可选：存储 report.html 供用户下载查看
```

## 8. 平台适配注意事项

### 8.1 run_id 获取

`run_id` 由引擎自动生成（格式 `YYYYMMDD-HHMMSS-<hex8>`），平台需要在 `report-dir` 下扫描最新目录获取：

```python
from pathlib import Path
import json

def get_latest_report(report_dir: str) -> dict:
    """获取最新一次执行的 JSON 报告。"""
    report_path = Path(report_dir)
    run_dirs = sorted(report_path.iterdir(), reverse=True)
    if not run_dirs:
        return None
    result_file = run_dirs[0] / "result.json"
    if result_file.exists():
        return json.loads(result_file.read_text(encoding="utf-8"))
    return None
```

或者更可靠的方式：指定唯一的 `report-dir` 给每次执行，执行后直接遍历该目录下唯一的子目录。

### 8.2 超时控制

平台应在 subprocess 层设置超时，防止用例执行无限挂起：

```python
try:
    result = subprocess.run(cmd, capture_output=True, text=True, timeout=600)
except subprocess.TimeoutExpired:
    # 标记为超时，清理进程
    pass
```

### 8.3 并发执行

当前引擎为单进程单用例执行模式。平台如需并发，应在平台调度层启动多个子进程，每个进程执行不同用例文件，使用不同的 `--report-dir` 避免冲突。

### 8.4 环境隔离

- 每次执行使用独立的 `--report-dir`，避免报告覆盖
- 通过 `--base-url` 切换目标环境（开发/测试/预发/生产）
- 通过 `--var` 注入环境相关变量（如 token、账号等）

### 8.5 用例文件管理

平台可选择以下方式管理用例文件：

1. **动态生成**：用户在 UI 编排后，平台实时生成 YAML 写入临时目录
2. **Git 仓库**：用例存储在 Git 仓库，平台 clone 后执行
3. **对象存储**：用例存储在 OSS/S3，执行前下载到本地

### 8.6 HTML 报告嵌入

HTML 报告为单文件、无外部依赖，可通过以下方式在平台中展示：

```html
<!-- iframe 嵌入 -->
<iframe src="/api/reports/{run_id}/report.html" width="100%" height="800px"></iframe>

<!-- 或提供下载链接 -->
<a href="/api/reports/{run_id}/report.html" download>下载 HTML 报告</a>
```

## 10. 完整接入示例（Python）

```python
"""测试平台接入 MwjRunner 的完整示例。"""

import json
import subprocess
import tempfile
from pathlib import Path

import yaml


def create_case_file(case_data: dict, work_dir: Path) -> Path:
    """将平台用例数据转换为 YAML 文件。"""
    case_file = work_dir / "case.yaml"
    case_file.write_text(yaml.dump(case_data, allow_unicode=True), encoding="utf-8")
    return case_file


def execute_case(case_file: Path, base_url: str, variables: dict = None) -> dict:
    """执行用例并返回结构化结果。"""
    report_dir = case_file.parent / "reports"
    cmd = [
        "mwjrunner", "run", str(case_file),
        "--base-url", base_url,
        "--report", "json,html",
        "--report-dir", str(report_dir),
    ]
    for key, value in (variables or {}).items():
        cmd.extend(["--var", f"{key}={value}"])

    proc = subprocess.run(cmd, capture_output=True, text=True, timeout=600)

    # 查找生成的报告
    report_dirs = sorted(report_dir.iterdir()) if report_dir.exists() else []
    result_data = None
    if report_dirs:
        result_file = report_dirs[0] / "result.json"
        if result_file.exists():
            result_data = json.loads(result_file.read_text(encoding="utf-8"))

    return {
        "exit_code": proc.returncode,
        "report": result_data,
        "html_path": str(report_dirs[0] / "report.html") if report_dirs else None,
    }


# 使用示例
if __name__ == "__main__":
    case = {
        "name": "健康检查",
        "steps": [{
            "name": "GET /health",
            "request": {"method": "GET", "url": "/health"},
            "assertions": [{"type": "status_code", "expected": 200}],
        }],
    }

    with tempfile.TemporaryDirectory() as tmp:
        work_dir = Path(tmp)
        case_file = create_case_file(case, work_dir)
        result = execute_case(case_file, base_url="http://localhost:8000")

        if result["exit_code"] == 0:
            print("PASSED")
        else:
            print(f"FAILED (exit_code={result['exit_code']})")
            if result["report"]:
                print(json.dumps(result["report"]["summary"], indent=2, ensure_ascii=False))
```

## 6. 认证接入

平台使用 JWT 双 Token 机制，所有业务 API 都需要认证。

### 6.1 登录获取 Token

```http
POST /api/auth/login
Content-Type: application/json

{
  "username": "admin",
  "password": "admin123"
}
```

响应：

```json
{
  "access_token": "eyJhbGciOiJIUzI1NiIs...",
  "refresh_token": "eyJhbGciOiJIUzI1NiIs...",
  "token_type": "bearer",
  "user": {
    "id": 1,
    "username": "admin",
    "role": "admin",
    "team_id": 1
  }
}
```

### 6.2 使用 Token

所有后续请求在 Header 中携带 Access Token：

```http
GET /api/cases
Authorization: Bearer eyJhbGciOiJIUzI1NiIs...
```

### 6.3 刷新 Token

Access Token 有效期 30 分钟，过期后使用 Refresh Token 刷新：

```http
POST /api/auth/refresh
Content-Type: application/json

{
  "refresh_token": "eyJhbGciOiJIUzI1NiIs..."
}
```

响应：

```json
{
  "access_token": "新的access_token...",
  "token_type": "bearer"
}
```

### 6.4 登出

```http
POST /api/auth/logout
Authorization: Bearer eyJhbGciOiJIUzI1NiIs...
```

登出后 Token 进入黑名单，无法再使用。

### 6.5 Token 集成建议

```python
import httpx

class MwjRunnerClient:
    def __init__(self, base_url: str, username: str, password: str):
        self.base_url = base_url
        self.client = httpx.Client(base_url=base_url)
        self._login(username, password)

    def _login(self, username: str, password: str):
        resp = self.client.post("/api/auth/login", json={
            "username": username,
            "password": password,
        })
        data = resp.json()
        self.access_token = data["access_token"]
        self.refresh_token = data["refresh_token"]
        self.client.headers["Authorization"] = f"Bearer {self.access_token}"

    def _refresh(self):
        resp = self.client.post("/api/auth/refresh", json={
            "refresh_token": self.refresh_token,
        })
        data = resp.json()
        self.access_token = data["access_token"]
        self.client.headers["Authorization"] = f"Bearer {self.access_token}"

    def request(self, method: str, path: str, **kwargs):
        resp = self.client.request(method, path, **kwargs)
        if resp.status_code == 401:
            self._refresh()
            resp = self.client.request(method, path, **kwargs)
        return resp
```

## 7. 权限模型

### 7.1 角色体系

| 角色 | 数据范围 | 操作权限 |
|------|----------|----------|
| admin | 全局（跨团队） | 所有操作 + 用户管理 |
| member | 本团队 | CRUD 本团队数据 |
| viewer | 本团队 | 只读 |

### 7.2 数据隔离

所有业务 API 自动按 `team_id` 过滤数据：
- 普通用户只能看到和操作本团队的用例、执行记录、环境等
- admin 角色可以跨团队访问所有数据
- 用户管理 API 仅 admin 可用

### 7.3 权限错误处理

| HTTP 状态码 | 含义 | 处理方式 |
|-------------|------|----------|
| 401 | Token 无效或过期 | 刷新 Token 或重新登录 |
| 403 | 权限不足 | 提示用户联系管理员 |

## 11. WebSocket 实时推送

### 11.1 连接

```
ws://host/ws?token=<access_token>
```

连接后服务端按团队隔离广播事件，不会收到其他团队的消息。

### 11.2 心跳

客户端定期发送 `ping` 文本消息，服务端回复：

```json
{"event": "pong"}
```

### 11.3 事件类型

| 事件 | 触发时机 | data 内容 |
|------|----------|-----------|
| `execution:created` | 执行创建时 | `{id, status, case_ids}` |
| `execution:update` | 执行完成时 | `{id, status, summary}` |

### 11.4 前端集成示例

```javascript
function connectWebSocket(token) {
  const ws = new WebSocket(`ws://${location.host}/ws?token=${token}`);

  ws.onmessage = (event) => {
    const msg = JSON.parse(event.data);
    if (msg.event === 'execution:update') {
      // 刷新执行列表或更新状态
      refreshExecutionList();
    }
  };

  // 心跳保活
  setInterval(() => {
    if (ws.readyState === WebSocket.OPEN) {
      ws.send('ping');
    }
  }, 30000);

  // 断线重连
  ws.onclose = () => {
    setTimeout(() => connectWebSocket(token), 3000);
  };
}
```

## 14. Webhook 触发


```http

Content-Type: application/json
```

```http
  "source": "jenkins"
}
```

适用于 Jenkins 或其他 CI 系统。

### 14.4 Webhook 配置流程

1. 在平台创建 Pipeline，记录 `pipeline_id`
2. 复制对应的 Webhook URL
3. 在代码仓库的 Webhook 设置中添加 URL
4. 配置签名密钥（GitHub）或 Token（GitLab）
5. 代码推送后自动触发 Pipeline 执行

## 18. 接入注意事项

### 18.1 错误码约定

| HTTP 状态码 | 含义 |
|-------------|------|
| 200 | 成功 |
| 201 | 创建成功 |
| 400 | 请求参数错误 |
| 401 | 未认证或 Token 过期 |
| 403 | 权限不足 |
| 404 | 资源不存在 |
| 422 | 请求体校验失败 |
| 500 | 服务器内部错误 |

### 18.2 分页约定

列表 API 支持分页参数：

```http
GET /api/cases?page=1&page_size=20
```

### 18.3 敏感信息

- 所有 API 响应中不会返回密码明文
- 日志和报告中 token、password、cookie、secret、authorization 等字段自动脱敏
- JWT Secret Key 不会通过 API 暴露

### 18.4 并发安全

- SQLite 模式下建议 gunicorn worker 数量 2-4
- 大规模并发场景建议迁移到 PostgreSQL
- WebSocket 连接数建议控制在单实例 1000 以内
