# T1-T8 审计修复设计

日期：2026-05-08

## 目标

对 MwjRunner 已完成的 T1-T8 做审计型修复，检查是否存在遗漏、技术债务或隐藏功能性问题，并修复属于 T1-T8 范围的问题。

## 范围

覆盖：

- T1：文档工作流和使用手册。
- T2：FastAPI 示例服务。
- T3：与示例服务联动的 YAML 示例用例。
- T4：M1 项目骨架和 CLI。
- T5：YAML 用例加载和校验。
- T6/T6R：HTTP 请求执行、结构化错误、快照、脱敏。
- T7：自研基础断言。
- T8：变量替换和 JSONPath 提取。

不覆盖：

- T9 Console 和 JSON 报告实现。
- HTML 报告、调度增强、并发、重试等后续任务。
- 与 T1-T8 无关的重构或依赖调整。

## 推荐方案

采用审计型修复：先按文档验收标准和现有实现逐项检查，再运行测试定位问题，最后只修复 T1-T8 范围内的遗漏、技术债和隐藏功能缺陷。

## 执行步骤

1. 对照 `doc/下一步计划.md`、`doc/需求规格说明书.md`、`doc/技术方案.md` 的 T1-T8 要求。
2. 阅读相关源码、示例 YAML、测试和使用手册。
3. 运行现有测试，记录失败和覆盖缺口。
4. 修复 T1-T8 范围内的问题，并补充必要测试。
5. 同步文档，确保计划、实现、示例和使用手册一致。
6. 使用项目 code-review 技能做提交前审查。
7. 精确暂存相关文件，提交并推送。

## 质量约束

- 不使用 pytest 作为 MwjRunner 核心执行框架或核心断言机制；pytest 仅用于项目自身测试。
- 不引入 Allure。
- 不新增与当前目标无关的依赖。
- HTTP、断言、变量、提取相关输出必须继续遵守敏感信息脱敏要求。
- 修复保持克制，不为了单一问题创建复杂抽象或未来预留代码。

## 验收标准

- T1-T8 文档、源码、示例和测试不存在已发现的明显遗漏。
- 相关单元测试和可执行验证通过，或对无法执行的验证给出原因。
- 代码审查无 P0/P1 阻塞问题。
- 变更已提交并推送到当前分支。
